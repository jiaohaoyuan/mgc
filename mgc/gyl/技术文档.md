## 1. 项目大局观

本项目“供应链决策平台（SCMP）”是一套极具现代化、响应迅速的前后端分离系统，兼顾传统ERP强数据校验（基础主数据）及互联网看板的动效交互场景。

### 技术定型
- **前端体系**：Vue 3 (Composition API) + TypeScript + Vite + Element Plus + Pinia + Vue Router + ECharts。
  - *极致优化*：采用了基于 `Keep-alive` 的路由内存封存技术解决多标签频繁切换卡顿；基于 `dynamic import` 按需懒加载沉重的 Excel ģ�飨`xlsx` 包），消除了页面初始挂载的阻塞。
- **后端体系**：Node.js + Express + MySQL + Redis。
  - *极致特性*：原生的 `multer` 聚合大通量 Excel 上传管道；原生手撸的防丢包 Upsert (插入或修改) 引擎；基于 Redis 和内置 HelperCode 的双重高规密码拦截。

### 核心功能象限
1. **组织权限基建模块** (系统库)：部门、用户、角色、职责岗位及路由视图动态关联底盘。
2. **基础主数据模块 (MDM)** (专设库)：针对核心商品域(SKU)、经销域、物流域的数据中心，支持批量带错误回滚的 Upsert 同步。
3. **数字智慧大屏**：整合地图与高频数据下发，包含智能订购演算图与牧场生产感知中心。

## 2. 数据库与领域模型划分

由于业务庞杂，数据库明确在物理层面/逻辑切面上分离为 **业务系统自身配置数据**（`cdop_sys`） 和 **纯粹的基础主数据中心**（`cdop_master`）。

### 2.1 业务平台库 (`cdop_sys`)
处理人员架构和运转权限：
- `t_ryytn_account`: 用户主表 (关键字段：密码通过非对称或其他方式对比，拥有手机号)
- `t_ryytn_oa_department` / `jobtitle`: 钉钉生态对应的自然部门结构和职级树
- `t_ryytn_role` / `t_ryytn_page`: RBAC关联基础角色资源点
- `t_ryytn_order_main`: 挂载在这里的高频订单系统等流转票据

### 2.2 主数据管理库 (`cdop_master`)
纯粹处理供应链基础数据（以 SKUs 为例，均带有严格复合主键、生效状态控制与时间索引）：
- `mst_sku_info`: SKU 商品域主实体表，带有69码、体积比率、保质期转换等物流信息。
- `mst_rltn_sku_reseller`: SKU 实体与经销商（代理）的商业售卖生效期强关联授权表。由于带生效结束期，故设计了特定的逻辑回算接口应对 Excel 录入。

> [!NOTE] 映射关系速查
所有 MySQL 的布尔表达基本为 TINYINT `1`(有效) 和 `0`(作废/逻辑删除/关闭)。在 Vue 前端的表格回填显示中被拦截转化为 `success` 或 `info/danger` 颜色 Tag。

## 3. 请求时序与安全拦截网

本期项目彻底告别了传统外包只追求业务而忽视安全防渗的逻辑缺陷：

### 3.1 “找回密码”与短信身份认证机制
不再允许仅凭“点击下一步”进行密码盲刷。系统拥有一个双因子锁：
1. **因子一（短信验证码）**：`POST /api/sms/send-code` 连线阿里云云市场-国阳云API下发6位动态码并加密储存至 Redis 的 `sms:code:{username}` 中（生命周期 5 分钟）。验证行为 `POST /api/sms/verify-code` 于后端严格查库比对验证码并检查手机号映射关系。
2. **因子二（辅助动态身份码）**：由于平台不对外公开，当企业员工手机号未收录时，可强制联系超级管理员。超管在平台顶栏点开动态码面板获取，并告知员工。动态码带有刷新失效（Refresh）、唯一拦截。

### 3.2 文件流与大数据导入防崩溃
后端位于 `server/index.js` 中的 `/api/master/import` ：
1. 使用 `multer` 以内存缓冲 (buffer) 形式硬接前端 Excel 对象，抛弃传统的落盘慢 IO 形式。
2. 将数据切片进入 `processRow` 通配逻辑器中验证。
3. 任何 Excel 内的数据违规（如结束时间早于开始时间），不会影响前后其他兄弟行的插入与全量覆盖更新 (`ON DUPLICATE KEY UPDATE`)，违规报错被数组化存底，直接透传给 Vue 前端的 Alert 面板，精准提示至 Excel 第 X 行出现什么问题。

本指南取代过往所有 STARTUP 文档，请确保您的 Windows/Linux 服务器上装有 Redis (端口 6379 留空) 和 MySQL (root/123456)。

**第一步：导入数据库环境**
如果您是去新电脑重新搭建，先在空 MySQL 连接中创建 `cdop_sys` 并导入 `cdop_sys_full_export_fixed.sql`，随后可手动建立 `cdop_master` 并执行新建语句（见项目中的 SQL）。

**第二步：唤醒双生服务进程**
由于前后端通过不同端口独立运转，您需启动两个终端分别挂载：

终端 A (打穿底层 API 与数据通信隧道)：
```bash
cd server
npm install
npm start
```
此时能看到 `[Backend API] Server is running on http://localhost:3000` 与 Redis 接入成功的提示。

终端 B (开启前端极速编译热更引擎)：
```bash
## 必须位于 gyl 根目录 ##
npm install
npm run dev
```

**第三步：访问系统**
通过 `http://localhost:5173` 访问您的专属门户。超管账号名一般为内置最高权限户或者数据库写入的管理员用户。如果您在页面修改任何源代码，由于 Vite 热引擎与我们新优化的 `<keep-alive>` 与大依赖拆包体系，页面会顺畅切片更新而不会卡顿白屏。

## 6. RBAC 并集鉴权实现（2026-04-08）

本次版本将权限模型升级为“用户多角色并集”，并在前后端建立一致的鉴权口径。

### 6.1 多角色并集规则
1. 一个用户可绑定多个角色（`t_ryytn_account_role`）。
2. 一个角色可绑定多个权限点（`t_ryytn_role_page -> t_ryytn_page`）。
3. 用户最终权限 = 所有关联角色权限的并集（`DISTINCT` 去重）。

核心 SQL（后端 `getUserRbacContext`）：

```sql
SELECT DISTINCT r.id, r.name
FROM t_ryytn_account_role ar
JOIN t_ryytn_role r ON ar.role_id = r.id
WHERE ar.account_id = ? AND (r.status = 1 OR r.status IS NULL);

SELECT DISTINCT p.id, p.path, p.permission
FROM t_ryytn_account_role ar
JOIN t_ryytn_role r ON ar.role_id = r.id
JOIN t_ryytn_role_page rp ON rp.role_id = r.id
JOIN t_ryytn_page p ON rp.page_id = p.id
WHERE ar.account_id = ? AND (r.status = 1 OR r.status IS NULL);
```

  - MDM 分组仅超级管理员可见（物理隐藏）。

## 8. 本地文件持久化改造（2026-04-09）

### 8.1 存储方式
- 后端改为纯本地 JSON 持久化，不再依赖 MySQL/Redis。
- 存储文件：`server/local-data/db.json`。
- 数据引擎：`server/localDb.js`（负责初始化、读写、自动落盘）。

### 8.2 初始化与持久化行为
1. 首次启动若不存在 `db.json`，系统自动生成一份业务化种子数据（组织、岗位、角色、用户、MDM、订单、日历等）。
2. 之后所有新增/编辑/删除操作都会实时写回 `db.json`。
3. 服务重启后直接加载该文件，数据不会丢失。

### 8.3 MDM 补齐的数据域
- 品类（`category`）：按“乳制品 > 液态奶 > 常温纯牛奶”分层初始化。
- 工厂（`factory`）：安吉液态奶、常山奶粉等实际名称。
- 仓库（`warehouse`）：总部仓、RDC、牧场前置仓、城市分拨仓。
- 渠道（`channel`）：线下/经销/KA 等层级。
- 经销商（`reseller`）：苏州联华、抖音旗舰店等。
- 组织（`org`）：总部、大区、业务单元。
- SKU 与关系表：`sku`、`reseller_relation`、`rltn_warehouse_sku`、`rltn_org_reseller`、`rltn_product_sku`。
- 业务日历（`calendar`）：2026 年全量日历数据，支持月视图与列表查询。
