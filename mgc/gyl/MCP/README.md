# MCP / 动态外部资料说明

## 这个目录是做什么的

这里不放项目本身的业务真相，而是说明：

- 什么时候需要用动态外部资料
- 动态资料优先看什么来源
- 用完之后怎么记录影响

## 什么叫“动态外部资料”

指仓库本地代码之外，会随时间变化或依赖外部环境的信息，例如：

- 官方框架文档
- 组件库最新行为
- 浏览器自动化运行时文档
- 安全、上传、接口标准的最新说明
- 明确要求“最新”“当前”“今天”的信息

## 什么时候应该用

以下场景建议使用动态外部资料，而不是只靠记忆：

- Vue / Vue Router / Vite / Element Plus 行为是否有版本差异
- Express / multer / xlsx 的安全或接口用法是否发生变化
- 浏览器自动化、运行时、构建工具相关问题
- 用户明确要求“查一下”“确认最新”
- 你对某个框架事实没有把握

## 什么时候不该用

以下场景应先以仓库本地代码为准：

- 这个项目的业务规则
- 奶制品品类结构
- SKU / SPU / 映射规范
- 权限路径约定
- 页面与 API 的真实对接关系
- 本地 JSON 数据结构

## 资料优先级

1. 仓库内代码与本地文档
2. 官方文档
3. 运行时/工具官方说明
4. 其他来源（只在必要时）

## 使用原则

- 动态资料是辅助判断，不替代本仓库的真实实现。
- 如果外部文档与仓库现状冲突，先记录冲突，再以当前项目约束为准处理。
- 高风险决定要说明“为什么采用外部资料建议，为什么不直接套官方示例”。

## 建议记录格式

当一次任务确实依赖了动态外部资料，最好在最终说明里带上这三点：

- 查了什么
- 为什么必须查
- 它影响了哪个实现决策

## 本仓库的特别提醒

这个项目当前的核心真相主要在本地：

- `src/`
- `server/`
- `server/local-data/db.json`
- MDM / SKU / 品类规则文件

所以多数任务里，先读仓库，再考虑 MCP，通常才是对的顺序。
