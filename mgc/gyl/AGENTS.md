# AGENTS.md

## 这个文件是做什么的

这个文件定义本仓库里“代码任务该怎么工作”。

任何代理、自动化工具或协作者，在开始修改代码前，都应先阅读：

1. `AGENTS.md`
2. `docs/context/PROJECT_CONTEXT.md`
3. `Skills/` 下与任务最相关的技能说明
4. `MCP/README.md`（当任务涉及动态外部资料时）

## 仓库定位

这是一个面向乳制品供应链的业务系统，不是纯演示项目。

系统覆盖：

- 组织与权限管理
- 主数据管理（MDM）
- 奶制品品类、SKU、SPU 管理
- 仓库、工厂、经销商与关系表
- 智能订货、订单闭环、库存运营
- 渠道经营、流程中心、管理驾驶舱、牧场业务

## 技术事实

- 前端：Vue 3 + TypeScript + Vite + Element Plus + Pinia + Vue Router
- 后端：Express
- 本地持久化：`server/local-data/db.json`
- 默认前端开发地址：`http://localhost:5173`
- 默认后端 API 地址：`http://localhost:3000/api`

## 工作原则

- 把它当成真实业务系统来改，不把它当成一次性页面。
- 优先做局部、可验证的小改动，不做顺手的大重构。
- 改动前先读入口和上下游，不凭感觉直接改。
- 任何涉及路由、权限、SKU/SPU、品类、导入导出、数据迁移的修改，都要看全链路影响。
- 不覆盖自己没做的改动，除非明确被要求。
- 配置、文档、代码、验证结果要互相对得上。

## 每次代码任务的标准流程

1. 先读任务相关上下文和入口文件。
2. 判断影响层级：
   - 页面
   - 路由
   - store
   - API
   - 本地持久化
   - 种子/迁移/校验
3. 用最小安全改动完成任务。
4. 跑对应验证。
5. 汇报改了什么、验证了什么、还剩什么风险。

## 必做验证

- 前端改动：运行 `npm run build`
- 后端或数据规则改动：运行 `cd server && npm test`
- 导航、权限、接口联通性改动：至少做一次针对性链路检查

## 高风险文件

这些文件改动前后都要更谨慎：

- `src/App.vue`
- `src/main.ts`
- `src/router/index.ts`
- `src/stores/appStore.ts`
- `server/index.js`
- `server/localDb.js`
- `server/dairyCategoryCatalog.js`
- `server/spuCatalog.js`
- `server/validateDairyCategoryData.js`

## 领域约束

- 品类体系必须限定在乳制品范围内。
- 新增或规范化 SKU 时，默认遵守统一七段式标准编码。
- SPU、SKU、品类、关系表、业务页引用必须保持联通。
- 超级管理员专属 MDM 页面和 RBAC 行为不能被顺手改坏。
- 如果修改了主数据编码规则，要同步检查导入、治理、业务引用和本地数据迁移。

## 文档约定

- `docs/context/` 放“项目是什么、边界是什么、哪些事实不能猜”。
- `Skills/` 放“某类任务在这个仓库里该怎么做”。
- `MCP/` 放“什么时候需要动态外部资料，怎么使用外部资料”。

## 注释约定

- 文档层可以用中文为主。
- 代码层只在业务规则、数据流关键点、容易误改处加简短中文注释。
- 注释必须跟代码一起维护，不能写成过时说明。
