# SKU编码制定规则与现状说明

## 一、文档目的

这份文档解决两件事：

1. 解释当前项目里已经存在的 SKU 编码规则到底是什么。
2. 给出一套后续可以正式执行的 SKU 制定规则，避免以后新增商品时一人一个写法。

这份文档适合：

- 管理层确认编码治理口径
- 主数据人员制定录入规范
- 产品经理整理字段规则
- 开发人员后续做系统校验
- 新人和实习生理解“SKU 为什么这样写”

---

## 二、一句话结论

当前系统里的 SKU 编码，并不是完全没有规则，而是已经形成了一套“英文语义片段拼接”的隐含规则。

它的大致写法是：

`SKU-品类或工艺-属性或口味或人群-规格-包装数`

例如：

- `SKU-UHT-250-12`
- `SKU-UHT-HIGHCAL-250-12`
- `SKU-YOG-CHILLED-STRAW-200-10`
- `SKU-POWDER-ADULT-HIGHPRO-800`

这套规则的优点是直观、容易看懂，但问题也很明显：字段位置不完全固定，单位不统一，属性码混用，系统没有强校验，长期不适合企业级治理。

因此，建议后续正式采用一套“半可读、强约束、可扩展”的 SKU 编码标准：

`SKU-品类码-工艺码-规格码-包装码-属性码-流水号`

标准示例：

- `SKU-UHT-UHT-250ML-12BX-PLN-001`
- `SKU-UHT-UHT-250ML-12BX-HCL-001`
- `SKU-YOG-CHL-200G-10CP-STR-001`
- `SKU-PWD-PWD-800G-01CN-ADT-001`

---

## 三、先把 SKU 讲明白

### 1. 什么是 SKU

SKU 可以理解成“最小销售单元编码”。

只要有任何一个会影响销售、库存、价格、授权、配送、结算、陈列、报表统计的因素发生变化，就应该拆成不同的 SKU。

例如下面这些情况，通常都应该是不同 SKU：

- 250 毫升和 1 升
- 12 盒装和 24 盒装
- 原味和草莓味
- 常温和低温
- 成人奶粉和儿童奶粉
- 正常款和高钙款

### 2. 什么不是 SKU

下面这些概念，不要和 SKU 混在一起：

- 条码：是扫码识别码，不等于 SKU
- 产品编码：通常表示一个产品家族或一个产品系列，不等于最小销售单元
- 批次号：表示某一批生产批次，不等于 SKU
- 仓库编码：表示库存地点，不等于 SKU
- 渠道编码：表示销售渠道，不等于 SKU
- 活动编码：表示促销活动，不等于 SKU

### 3. 业务上为什么 SKU 很重要

当前项目里，SKU 不是一个孤立字段，而是很多业务的主连接键。

SKU 已经直接参与：

- 主数据管理
- 订单行
- 库存台账
- 库存锁定
- 仓库-SKU关系
- 产品-SKU关系
- SKU-经销商授权关系
- 渠道经营分析

所以 SKU 一旦规则混乱，后面会连带影响：

- 下单准确性
- 库存准确性
- 授权准确性
- 价格口径
- 报表统计
- 数据迁移和对账

## 2. 当前规则的实际结构

把现有样本拆开看，当前隐含规则大致如下：

`SKU-业务类别-细分类或属性-规格-包装数`

但要注意，这只是“大致规律”，不是严格固定规则。

因为当前系统里有些编码只有 4 段，有些有 5 段，有些有 6 段。

### 3. 现有规则的拆解示例

| 现有编码 | 可以理解成什么 |
| --- | --- |
| `SKU-UHT-250-12` | 常温奶，250 毫升，12 盒装 |
| `SKU-UHT-HIGHCAL-250-12` | 常温奶，高钙，250 毫升，12 盒装 |
| `SKU-UHT-A2-250-12` | 常温奶，A2，250 毫升，12 盒装 |
| `SKU-PASTEUR-950` | 巴氏鲜奶，950 毫升，单瓶或默认单件 |
| `SKU-PASTEUR-CHILD-750` | 巴氏鲜奶，儿童款，750 毫升 |
| `SKU-YOG-CHILLED-STRAW-200-10` | 酸奶，低温，草莓味，200 克，10 杯装 |
| `SKU-YOG-DRINK-MANGO-330` | 风味酸奶饮品，芒果味，330 毫升 |
| `SKU-POWDER-ADULT-HIGHPRO-800` | 奶粉，成人，高蛋白，800 克 |
| `SKU-CHEESE-MOZ-200` | 芝士，马苏里拉，200 克 |

### 4. 当前规则里每一段通常表示什么

#### 第一段：固定前缀

固定写 `SKU`，表示这是商品最小销售单元编码。

#### 第二段：大类或工艺

常见值：

- `UHT`：常温灭菌奶
- `PASTEUR`：巴氏鲜奶
- `YOG`：酸奶
- `PROBIOTIC`：益生菌饮品
- `POWDER`：奶粉
- `CHEESE`：芝士

#### 第三段及之后：细分类、属性、适用人群、风味等

常见值：

- `HIGHCAL`：高钙
- `A2`：A2
- `ORGANIC`：有机
- `CHILD`：儿童
- `ADULT`：成人
- `MIDDLE`：中老年
- `CHILLED`：低温
- `DRINK`：饮品型
- `GREEK`：希腊型
- `ORIGIN`：原味
- `STRAW`：草莓味
- `BLUEB`：蓝莓味
- `0SUGAR`：零蔗糖
- `MANGO`：芒果味
- `MOZ`：马苏里拉
- `CREAM`：奶油芝士
- `SLICE`：奶酪片
- `HIGHPRO`：高蛋白

#### 倒数第二段：规格

常见值：

- `250`
- `200`
- `330`
- `700`
- `800`
- `950`
- `1L`

这里的问题是：有些带单位，有些不带单位。

例如：

- `1L` 带单位
- `250` 没写单位，但业务上默认理解成毫升
- `200` 在酸奶和芝士里又可能默认理解成克

#### 最后一段：包装数

有些 SKU 最后一段是包装数，例如：

- `12`
- `24`
- `10`
- `30`

但并不是所有 SKU 都有这一段。

例如：

- `SKU-PASTEUR-950`
- `SKU-CHEESE-MOZ-200`

这说明当前编码里“包装数”并没有完全统一。

## 五、当前系统为什么会形成这样的 SKU 规则

这是一个非常典型的“演示期到试运行期”编码习惯。

通常项目在早期会优先追求：

- 先有数据能跑起来
- 先让页面能展示
- 先让业务同学看懂
- 先让测试和演示顺手

所以最自然的做法就是：

把大家最关心的商品特征，直接拼进 SKU 编码里。

这样做的好处是：

- 一眼能看出大概是什么商品
- 下单时容易选
- 库存和订单演示时容易讲
- 不需要额外查字典

这就是当前规则存在的业务合理性。

所以不能说它“错”，而应该说它“适合早期，但不适合长期治理”。

---

## 六、当前 SKU 规则的优点

### 1. 可读性强

看到编码就能大概知道商品是什么。

### 2. 沟通效率高

业务人员在讨论时，不用每次都点开商品详情。

### 3. 适合演示和试运行

在 SKU 数量不多的时候，这种写法成本低、理解快。

### 4. 有一定分类能力

项目里已经有代码直接通过前缀判断商品大类，例如：

- `SKU-UHT-`
- `SKU-PASTEUR-`
- `SKU-YOG-`
- `SKU-POWDER-`
- `SKU-CHEESE-`

这说明现有编码已经被部分业务逻辑实际使用。

---

## 七、当前 SKU 规则的主要问题

### 1. 规则是“默契”，不是“制度”

现在系统里对 SKU 的校验，主要只有两条：

- 不能为空
- 不能重复

但没有强制规定：

- 一共必须几段
- 每一段分别代表什么
- 哪些词允许出现
- 单位必须怎么写
- 包装数必须怎么写

这意味着只要有人愿意，他完全可以新增一个：

- `SKU-TEST-01`
- `SKU-DEMO01`
- `SKU-MILK-A`

系统也可能放进去。

### 2. 段位不固定

例如：

- `SKU-UHT-250-12` 是 4 段
- `SKU-UHT-HIGHCAL-250-12` 是 5 段
- `SKU-YOG-CHILLED-STRAW-200-10` 是 6 段

段位不固定，后面做自动拆解、自动统计、自动校验都会麻烦。

### 3. 单位表达不统一

当前同时存在：

- `250`
- `200`
- `950`
- `1L`

问题在于：

- `250` 到底是毫升还是克，靠人猜
- `1L` 带单位，`950` 却不带单位

长期看，这种写法不利于标准化。

### 4. 属性码混在一起

当前编码里的中间片段，混合了很多不同性质的信息：

- 品类细分
- 功能卖点
- 适用人群
- 风味口味
- 温层
- 产品形态

例如：

- `HIGHCAL`
- `A2`
- `ORGANIC`
- `CHILD`
- `MANGO`
- `CHILLED`
- `DRINK`

这些概念属于不同维度，但现在都用同一种“自由片段”拼进去，后面很容易失控。

### 5. 不利于未来扩展

如果后面要继续扩展到：

- 品牌
- 子品牌
- 礼盒装
- 联名款
- 电商专供
- 节日限定
- 不同配方版本

继续沿用现在的“想到什么拼什么”的方式，SKU 会越来越长，也越来越乱。

### 6. 不利于主数据治理

当前 SKU 已经和订单、库存、经销商授权、仓库覆盖等数据强关联。

如果编码继续无标准扩展，就会出现：

- 同类商品命名口径不一致
- 导入模板填写不统一
- 人工新增容易出错
- 统计维度难拆分
- 数据迁移映射复杂

## 八、建议正式采用的新 SKU 制定规则

## 1. 设计原则

正式的 SKU 编码规则，建议遵守下面六个原则。

### 原则一：稳定

一旦建档，尽量不要因为名称优化、营销文案变化而改 SKU。

### 原则二：可读

看到编码后，业务人员至少能判断大类、规格、包装和主要属性。

### 原则三：可校验

系统可以通过固定格式校验新增 SKU 是否合规。

### 原则四：可扩展

未来增加新品类、新规格、新属性时，不需要推翻整套规则。

### 原则五：不混入动态信息

SKU 不要编码以下动态内容：

- 仓库
- 渠道
- 大区
- 活动
- 批次
- 日期
- 库龄
- 价格等级
- 客户

### 原则六：一物一码

只要销售单元不同，就必须是不同 SKU。

## 九、推荐的正式编码结构

建议正式采用下面这个结构：

`SKU-品类码-工艺码-规格码-包装码-属性码-流水号`

### 1. 各段含义

| 段位 | 含义 | 说明 |
| --- | --- | --- |
| 第 1 段 | `SKU` | 固定前缀 |
| 第 2 段 | 品类码 | 表示大类 |
| 第 3 段 | 工艺码 | 表示工艺或产品形态 |
| 第 4 段 | 规格码 | 必须带单位 |
| 第 5 段 | 包装码 | 必须标准化 |
| 第 6 段 | 属性码 | 主要差异化属性 |
| 第 7 段 | 流水号 | 当前面组合仍需细分时使用 |

### 2. 为什么这样设计

这样设计的好处是：

- 前三段解决“这是什么商品”
- 第四段解决“多大规格”
- 第五段解决“怎么装”
- 第六段解决“和同类相比有什么关键差异”
- 第七段解决“同组合下还需继续区分时如何唯一化”

## 十、推荐的字典口径

下面给出建议字典。后续如果正式落地，可以在主数据里维护一张“SKU 编码字典表”。

## 1. 品类码建议

| 品类码 | 中文含义 |
| --- | --- |
| `UHT` | 常温液奶 |
| `FRM` | 巴氏鲜奶 |
| `YOG` | 酸奶 |
| `PWD` | 奶粉 |
| `CHS` | 芝士 |
| `DRK` | 乳饮品 |

说明：

- 当前系统里的 `PASTEUR` 建议以后收敛为更短的固定品类码，例如 `FRM`
- 当前系统里的 `PROBIOTIC` 建议不要直接作为大类，最好归到乳饮品或酸奶品类中

## 2. 工艺码建议

| 工艺码 | 中文含义 |
| --- | --- |
| `UHT` | 超高温灭菌 |
| `PAS` | 巴氏杀菌 |
| `CHL` | 冷藏低温 |
| `RTD` | 即饮型 |
| `GRK` | 希腊型 |
| `PWD` | 粉状 |
| `SLC` | 切片型 |
| `CRM` | 奶油型 |
| `MOZ` | 马苏里拉型 |

说明：

- 这里的“工艺码”不是只写工艺，也可以承载产品形态
- 关键不是名字，而是全公司以后只用一套字典

## 3. 规格码建议

规格码必须带单位，统一写法如下：

| 示例 | 中文含义 |
| --- | --- |
| `250ML` | 250 毫升 |
| `950ML` | 950 毫升 |
| `1L` | 1 升 |
| `200G` | 200 克 |
| `800G` | 800 克 |

规则要求：

- 液体统一用 `ML` 或 `L`
- 粉剂、固体统一用 `G` 或 `KG`
- 不允许只写数字不写单位

## 4. 包装码建议

| 包装码 | 中文含义 |
| --- | --- |
| `01BT` | 单瓶 |
| `01BX` | 单盒 |
| `01BG` | 单袋 |
| `01CN` | 单罐 |
| `10CP` | 10 杯装 |
| `12BX` | 12 盒装 |
| `24BX` | 24 盒装 |
| `30BT` | 30 瓶装 |

规则要求：

- 前两位写数量
- 后两位写包装类型
- 数量不足两位前面补零

例如：

- `1` 盒不要写 `1BX`
- 应写成 `01BX`

## 5. 属性码建议

属性码只保留“最核心、最稳定、最有区分价值”的那一个主属性。

| 属性码 | 中文含义 |
| --- | --- |
| `PLN` | 基础款或原味 |
| `HCL` | 高钙 |
| `A2N` | A2 |
| `ORG` | 有机 |
| `ADT` | 成人 |
| `CHD` | 儿童 |
| `MID` | 中老年 |
| `STR` | 草莓 |
| `BLU` | 蓝莓 |
| `MNG` | 芒果 |
| `ZSG` | 零蔗糖 |

规则要求：

- 一个 SKU 只保留一个主属性码
- 其他详细卖点放到商品名称和扩展属性里，不要无限塞进 SKU 编码

## 6. 流水号建议

流水号用于兜底唯一性。

标准写法：

- `001`
- `002`
- `003`

使用原则：

- 如果前面的组合已经足够唯一，也建议保留流水号，便于未来扩展
- 流水号不要跳着写，不要带业务含义

## 十二、为什么推荐这套规则，而不是继续沿用现有规则

### 1. 现有规则更像“描述型编码”

它适合少量商品阶段。

### 2. 新规则更像“治理型编码”

它适合：

- 企业规模扩大
- 商品数量增加
- 多部门协同
- 系统自动校验
- 未来切换到正式数据库
- 做主数据审批和版本管理

### 3. 新规则兼顾了两件事

- 业务还能大致看懂
- 系统也能稳定校验

这比完全纯流水号的黑盒编码更适合当前项目阶段。

## 十三、如何操作：以后新增一个 SKU 时，应该怎么定编码

下面给出标准操作步骤。

## 第一步：先确认它是不是一个新的 SKU

先问自己下面几个问题：

1. 规格变了吗
2. 包装数变了吗
3. 温层变了吗
4. 工艺变了吗
5. 口味变了吗
6. 核心功能卖点变了吗
7. 适用人群变了吗

只要有一个关键销售属性变化，通常就应该建新 SKU。

## 第二步：确定品类码

先判断它属于：

- 常温液奶
- 巴氏鲜奶
- 酸奶
- 乳饮品
- 奶粉
- 芝士

然后从字典表里选固定品类码。

## 第三步：确定工艺码

判断它的主要工艺或形态，例如：

- 超高温灭菌
- 巴氏杀菌
- 冷藏低温
- 即饮型
- 希腊型
- 粉状
- 奶油型

## 第四步：确定规格码

把净含量转换成标准写法：

- 250 毫升写成 `250ML`
- 950 毫升写成 `950ML`
- 200 克写成 `200G`
- 800 克写成 `800G`

## 第五步：确定包装码

例如：

- 单瓶：`01BT`
- 单盒：`01BX`
- 单罐：`01CN`
- 12 盒装：`12BX`
- 10 杯装：`10CP`

## 第六步：确定主属性码

只选一个最重要的主属性。

例如：

- 原味或基础款：`PLN`
- 高钙：`HCL`
- 草莓：`STR`
- 儿童：`CHD`

如果一个商品同时有多个卖点，例如“儿童高钙草莓酸奶”，不要把所有信息都塞进 SKU。

应遵守一个原则：

SKU 只保留主差异属性，剩余信息放到商品名称和扩展字段。

## 第七步：分配流水号

如果这个组合下是第一个商品，就写：

- `001`

如果后面又出了配方调整版或另一个同组合商品，再往后排：

- `002`
- `003`

## 第八步：形成最终编码

把前面所有片段按顺序拼起来：

`SKU-品类码-工艺码-规格码-包装码-属性码-流水号`

例如：

“低温草莓酸奶 200 克 10 杯装”

拆分为：

- 品类码：`YOG`
- 工艺码：`CHL`
- 规格码：`200G`
- 包装码：`10CP`
- 属性码：`STR`
- 流水号：`001`

最终编码：

`SKU-YOG-CHL-200G-10CP-STR-001`

---

## 十四、正例与反例

## 1. 正例

- `SKU-UHT-UHT-250ML-12BX-PLN-001`
- `SKU-FRM-PAS-950ML-01BT-PLN-001`
- `SKU-YOG-CHL-200G-10CP-STR-001`
- `SKU-PWD-PWD-800G-01CN-ADT-001`

这些编码的特点是：

- 段位固定
- 单位明确
- 包装明确
- 属性明确
- 便于系统校验

## 2. 反例

- `SKU-DEMO01`
- `SKU-MILK-A`
- `SKU-250ML`
- `SKU-UHT-TEST`
- `SKU-YOG-CHILLED-STRAWBERRY-BIGPACK-NEW`

这些编码的问题包括：

- 信息不完整
- 不符合固定段位
- 含义不标准
- 不利于维护
- 难以自动校验

## 十五、现有编码如何映射到新规则

为了避免系统一次性大改，建议采用“旧码保留、建立映射、逐步切换”的方式。

### 1. 旧码与新码映射示例

| 现有编码 | 推荐新编码 | 中文解释 |
| --- | --- | --- |
| `SKU-UHT-250-12` | `SKU-UHT-UHT-250ML-12BX-PLN-001` | 常温纯牛奶 250 毫升 12 盒装 |
| `SKU-UHT-HIGHCAL-250-12` | `SKU-UHT-UHT-250ML-12BX-HCL-001` | 高钙常温纯牛奶 |
| `SKU-UHT-A2-250-12` | `SKU-UHT-UHT-250ML-12BX-A2N-001` | A2 常温纯牛奶 |
| `SKU-PASTEUR-950` | `SKU-FRM-PAS-950ML-01BT-PLN-001` | 巴氏鲜奶 950 毫升 |
| `SKU-PASTEUR-CHILD-750` | `SKU-FRM-PAS-750ML-01BT-CHD-001` | 儿童鲜奶 750 毫升 |
| `SKU-YOG-CHILLED-ORIGIN-200-10` | `SKU-YOG-CHL-200G-10CP-PLN-001` | 低温原味酸奶 200 克 10 杯装 |
| `SKU-YOG-CHILLED-STRAW-200-10` | `SKU-YOG-CHL-200G-10CP-STR-001` | 低温草莓酸奶 |
| `SKU-YOG-CHILLED-BLUEB-200-10` | `SKU-YOG-CHL-200G-10CP-BLU-001` | 低温蓝莓酸奶 |

### 2. 迁移建议

建议不要立刻把所有旧编码直接替换掉。

更稳妥的方式是：

1. 保留原有 `sku_code` 作为当前运行编码
2. 新增一个“标准 SKU 编码”字段或映射表
3. 新增商品从新规则开始
4. 存量商品逐步建立映射
5. 待上下游接口、库存、订单、授权、报表全部适配后，再考虑切主编码

## 十六、系统后续应该补的校验规则

如果后面系统要正式落地这个规则，建议补下面几类校验。

### 1. 格式校验

要求：

- 固定 7 段
- 只能使用大写字母、数字、短横线
- 不允许空格
- 不允许中文

### 2. 字典校验

要求：

- 品类码必须在字典表里
- 工艺码必须在字典表里
- 包装码必须在字典表里
- 属性码必须在字典表里

### 3. 规格校验

要求：

- 必须带单位
- 液体只能用 `ML` 或 `L`
- 固体只能用 `G` 或 `KG`

### 4. 唯一性校验

要求：

- 新旧编码都要唯一
- 同一个标准组合不能重复建档

### 5. 业务一致性校验

要求：

- `SKU名称` 和 `SKU编码` 不能明显冲突
- 例如编码写 `250ML`，名称却写 `200ml`，应阻止保存

## 十七、给管理层的建议

从企业管理角度，SKU 编码不要再停留在“大家差不多会写”的阶段。

建议尽快做三件事：

### 1. 先定制度

把这份规则作为主数据标准发出去。

### 2. 再定字典

把品类码、工艺码、包装码、属性码沉淀成正式字典。

### 3. 最后再做系统校验

让系统强制执行规则，而不是继续靠人工记忆。

---

## 十八、最终建议

综合当前项目阶段和未来扩展需要，建议明确以下结论：

### 结论一

当前系统现有 SKU 规则可以概括为：

`SKU-品类或工艺-属性或细分类-规格-包装数`

它已经能支撑演示和局部试运行，但还不够标准。

### 结论二

后续正式标准建议统一为：

`SKU-品类码-工艺码-规格码-包装码-属性码-流水号`

### 结论三

今后新增 SKU，必须做到：

- 段位固定
- 单位统一
- 字典化管理
- 禁止自由发挥

### 结论四

不要把下面这些信息写进 SKU：

- 仓库
- 渠道
- 客户
- 价格
- 活动
- 批次
- 日期
- 地区

这些应该由别的数据对象来表达。

公司后续统一采用 `SKU-品类码-工艺码-规格码-包装码-属性码-流水号` 的七段式编码规则，所有新增 SKU 必须通过字典和格式校验，规格必须带单位，包装必须标准化，SKU 中禁止写渠道、仓库、活动、批次等动态信息。
