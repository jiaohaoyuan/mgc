# 渠道需求计划分阶段开发文档

更新时间：2026-04-28  
适用范围：渠道需求计划、产品锁定期、滚动版本、版本对比、Excel 下载  
当前存储策略：先使用 `server/local-data/db.json`，通过 `server/localDb.js` 读写；后续再迁移到 MySQL 与 Redis

---

## 1. 文档目标

这份文档把《渠道需求计划与产品锁定期开发规划》拆成可执行的阶段路线。

本阶段开发原则：

- 先形成可用闭环，再补增强能力。
- 先沿用本地 JSON 持久化，不直接接 MySQL。
- 后端接口、前端页面、权限、数据结构同步推进。
- 每个阶段完成后都要能单独验证。
- 不改 SKU 七段式编码规则，不破坏 MDM 超级管理员专属页面。

---

## 2. 总体开发顺序

推荐按 6 个阶段推进：

1. 阶段 0：准备与边界确认
2. 阶段 1：后端本地数据结构与基础接口
3. 阶段 2：前端页面、路由、菜单与计划版本基础操作
4. 阶段 3：版本编辑、渠道提交、整体确认闭环
5. 阶段 4：产品锁定期
6. 阶段 5：滚动版本、版本对比、Excel 下载
7. 阶段 6：联通回归与数据库化预留

阶段 1 到阶段 3 完成后，业务上已经可以完成一次“创建计划 -> 创建版本 -> 填报 -> 提交 -> 确认”的主流程。

## 4. 阶段 1：后端本地数据结构与基础接口

### 4.1 阶段目标

建立渠道需求计划后端模块，让计划、版本和基础选项能通过接口读写。

### 4.2 涉及文件

新增：

- `server/channelDemandPlan.js`

修改：

- `server/index.js`
- `server/localDb.js`

### 4.3 本地存储结构

在 `db.biz` 下补齐以下数组：

```json
{
  "channel_demand_plans": [],
  "channel_demand_plan_channels": [],
  "channel_demand_plan_skus": [],
  "channel_demand_plan_versions": [],
  "channel_demand_plan_channel_statuses": [],
  "channel_demand_plan_data": [],
  "product_lock_rules": [],
  "downstream_demand_plan_jobs": []
}
```

### 4.4 后端模块职责

`server/channelDemandPlan.js` 负责：

- `ensureChannelDemandPlanStructures(db)`：补齐业务数组。
- `registerChannelDemandPlanRoutes(options)`：注册全部路由。
- 周序列工具：解析 `2026W18`、生成连续周、跨年进位。
- 范围展开工具：按计划配置展开二级渠道和 SKU。
- 状态工具：计划状态、版本状态、渠道提交状态流转。

### 4.5 基础接口

实现以下接口：

- `GET /api/demand/channel-plan/options`
- `GET /api/demand/channel-plan`
- `POST /api/demand/channel-plan`
- `PUT /api/demand/channel-plan/:planCode`
- `DELETE /api/demand/channel-plan/:planCode`
- `GET /api/demand/channel-plan/:planCode/version`
- `POST /api/demand/channel-plan/:planCode/version`
- `GET /api/demand/channel-plan/version/:versionCode`

### 4.6 关键规则

新增计划：

- `plan_code` 格式：`CDP-YYYYMMDD-####`
- `week_count` 默认 8，最小 1，建议最大 26。
- `channel_scope=1` 时不写范围表，打开版本时按全部二级渠道展开。
- `channel_scope=2` 时写入 `channel_demand_plan_channels`。
- `sku_scope=1` 时不写范围表，打开版本时按全部有效 SKU 展开。
- `sku_scope=2` 时写入 `channel_demand_plan_skus`。

新建版本：

- `version_code` 同计划下唯一。
- 生成周列表。
- 为每个 `渠道 × SKU × 周` 初始化一行数据。
- 首版 `plan_value=null`。
- 有 `last_version_code` 时继承重叠周数据。
- 初始化每个渠道的提交状态。

### 4.7 验证

运行：

```bash
cd server && npm test
```

手动接口验证：

- 登录后请求 `GET /api/demand/channel-plan/options`，应返回二级渠道和 SKU。
- 新增计划后，`server/local-data/db.json` 的 `biz.channel_demand_plans` 出现记录。
- 新建版本后，版本表、渠道状态表、数据表均出现对应记录。

## 6. 阶段 3：版本编辑、渠道提交、整体确认闭环

### 6.1 阶段目标

实现最核心的业务闭环：按渠道填报 SKU 周度需求、提交渠道、整体确认。

### 6.2 涉及文件

修改：

- `server/channelDemandPlan.js`
- `src/views/ChannelDemandPlan.vue`

### 6.3 后端接口

实现：

- `GET /api/demand/channel-plan/version/:versionCode/data`
- `PUT /api/demand/channel-plan/version/:versionCode/data`
- `PUT /api/demand/channel-plan/version/:versionCode/submit`
- `PUT /api/demand/channel-plan/version/:versionCode/withdraw`
- `PUT /api/demand/channel-plan/version/:versionCode/confirm`

### 6.4 编辑表格

表格规则：

- 顶部使用二级渠道标签切换。
- 行为 SKU。
- 列为财年周。
- 末列为当前 SKU 合计。
- `plan_value=null` 显示为空。
- `plan_value=0` 显示为 0。
- 当前页只加载当前渠道数据。
- 保存时只提交变更格，减少 JSON 写入量。

### 6.5 渠道提交规则

提交本渠道时：

- 如果存在空值，后端返回空值数量。
- 前端弹出二次确认：“存在 N 个未填写数据，是否继续提交？”
- 用户确认后带 `allowEmpty=true` 再提交。
- 提交后该渠道所有格子只读。
- 所有渠道提交后，版本状态自动变为待确认。

### 6.6 管理员撤回规则

撤回条件：

- 仅超级管理员可撤回。
- 版本为草稿或提交中时可撤回。
- 已确认版本不可撤回。

撤回效果：

- 渠道提交状态变为未提交。
- 版本状态回退为提交中或草稿。
- 当前渠道恢复可编辑。

### 6.7 整体确认规则

确认前：

- 后端返回汇总预览。
- 汇总包含渠道合计、SKU 合计、空值数量。
- 前端展示确认弹窗。

确认后：

- 版本状态变为已确认。
- 全部数据只读。
- 写入 `db.biz.downstream_demand_plan_jobs`。
- 页面出现“下载 Excel”按钮，但导出功能可在阶段 5 完成。

### 6.8 验证

运行：

```bash
npm run build
cd server && npm test
```

手动链路：

1. 新建计划。
2. 新建首版。
3. 切换到第一个渠道。
4. 填写多个 SKU、多周数据。
5. 保存后刷新页面，数据不丢失。
6. 提交本渠道。
7. 检查该渠道只读。
8. 所有渠道提交后，版本进入待确认。
9. 整体确认后，版本进入已确认。

## 7. 阶段 4：产品锁定期

### 7.1 阶段目标

实现 SKU 在指定渠道、指定日期范围内的锁定规则，并控制计划格子的编辑权限。

### 7.2 涉及文件

修改：

- `server/channelDemandPlan.js`
- `src/views/ChannelDemandPlan.vue`

### 7.3 后端接口

实现：

- `GET /api/demand/channel-plan/product-lock-rules`
- `POST /api/demand/channel-plan/product-lock-rules`
- `PUT /api/demand/channel-plan/product-lock-rules/:id`
- `DELETE /api/demand/channel-plan/product-lock-rules/:id`
- `POST /api/demand/channel-plan/version/:versionCode/rebuild-locks`

### 7.4 锁定规则

一条锁定规则：

- SKU
- 多个二级渠道
- 开始日期
- 结束日期
- 备注

状态派生：

- 当前日期小于开始日期：未生效，可编辑可删除。
- 当前日期在起止日期内：生效中，不可删除。
- 当前日期大于结束日期：已过期，只读。

冲突校验：

- 同一 SKU + 同一渠道，日期范围不能重叠。
- 编辑时需要排除当前规则自身。

### 7.5 版本格子锁定

创建版本时：

- 根据锁定规则写入 `is_locked` 快照。
- 如果计划周日期范围与锁定规则日期范围有交集，则锁定。

刷新快照：

- 仅未确认版本允许刷新。
- 仅超级管理员可刷新。
- 已确认版本保持历史事实，不刷新。

编辑规则：

- 普通用户不能编辑锁定格。
- 超级管理员可以强制编辑锁定格。
- 强制编辑必须二次确认并填写原因。

### 7.6 验证

运行：

```bash
npm run build
cd server && npm test
```

手动验证：

- 新增未生效锁定规则，可编辑可删除。
- 新增当前生效锁定规则，不可删除。
- 同 SKU + 同渠道重叠规则新增失败。
- 新建版本后命中规则的格子灰底只读。
- 普通用户保存锁定格失败。
- 超级管理员强制修改锁定格成功，且记录原因。

## 8. 阶段 5：滚动版本、版本对比、Excel 下载

### 8.1 阶段目标

补齐 PRD 中的增强能力，让计划具备周度滚动、版本差异分析和对外下载能力。

### 8.2 涉及文件

修改：

- `server/channelDemandPlan.js`
- `src/views/ChannelDemandPlan.vue`

### 8.3 滚动版本

接口：

- `POST /api/demand/channel-plan/:planCode/version/roll`

规则：

- 取上一版本起始周 + 1 周作为新版本起始周。
- 按计划 `week_count` 生成新周列表。
- 重叠周继承上一版本数据。
- 新滚入周 `plan_value=null`。
- 重新计算锁定快照。
- 新版本状态为草稿。

本阶段不注册真实 Quartz。

### 8.4 版本对比

接口：

- `GET /api/demand/channel-plan/version/compare`

参数：

- `baseVersionCode`
- `targetVersionCode`
- `channelCode`

规则：

- 支持同计划下两个版本对比。
- 周列表取并集。
- SKU 列表取并集。
- 同位置数值不同标记为差异。
- 覆盖周期不一致时标记来源缺失。

### 8.5 Excel 下载

接口：

- `GET /api/demand/channel-plan/version/:versionCode/export`

规则：

- 仅已确认版本允许下载。
- 空值输出为 0。
- 文件名：`渠道需求计划_{plan_code}_{version_code}_{yyyyMMdd}.xlsx`。
- 字段包含计划编号、版本号、二级渠道编码/名称、SKU 编码/名称、财年周、需求数量。

### 8.6 验证

运行：

```bash
npm run build
cd server && npm test
```

手动验证：

- 滚动版本能继承重叠周。
- 新滚入周为空。
- 对比页面能高亮差异。
- 已确认版本能下载 Excel。
- 未确认版本下载被拒绝。

- 后续迁移到 MySQL 时，接口路径和前端调用不变。
