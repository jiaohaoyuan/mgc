# 渠道需求计划与产品锁定期开发规划

## 1. 规划目标

根据《渠道需求计划 PRD v1.1》和《产品锁定期功能》两个需求文档，在现有 Vue 3 + Express + 本地 JSON 持久化架构中补齐“渠道需求计划”能力。

本阶段不接入真实数据库，不创建 MySQL 表。PRD 中的表结构先映射为 `server/local-data/db.json` 下的业务数组，并通过 `server/localDb.js` 的 `readDb`、`updateDb`、`nextId`、`nowIso` 方式读写。

核心目标：

- 支持计划、版本、渠道提交状态、计划明细数据的完整维护。
- 支持产品锁定期规则，并在版本数据中生成 `is_locked` 快照。
- 支持普通用户查看锁定格、管理员二次确认后强制修改。
- 支持渠道提交、撤回、整体确认、版本对比、Excel 下载。
- 支持滚动版本创建，但本阶段先用“手动触发滚动生成”模拟定时任务。
- 支持整体确认后模拟触发下游分仓计划，不接真实 MQ。

## 2. 已阅读需求摘要

### 2.1 渠道需求计划

渠道需求计划用于维护“未来 N 周，每个二级渠道、每个 SKU 的预计需求数量”。

关键规则：

- 计划是长期对象，版本是每周滚动产生的一份计划表。
- 默认覆盖 8 周，计划粒度为“二级渠道 × SKU × 财年周”。
- 首次版本的 `plan_value` 为空，滚动版本继承重叠周，新滚入周为空。
- `NULL` 和 `0` 必须区分：`NULL` 是未填写，`0` 是明确零需求。
- 每个渠道可单独提交，提交后该渠道锁定。
- 全部渠道提交后版本自动进入“待确认”。
- 整体确认后版本进入“已确认”，全部数据锁定，并允许下载 Excel。
- 管理员可撤回草稿/提交中版本里的已提交渠道，已确认版本不可回退。

### 2.2 产品锁定期

产品锁定期是计划编辑权限控制规则。

一条锁定记录包含：

- SKU
- 指定渠道列表
- 开始日期
- 结束日期

规则：

- 当前日期在 `[开始日期, 结束日期]` 内：生效中，不可删除。
- 当前日期小于开始日期：未生效，可编辑可删除。
- 当前日期大于结束日期：已过期，只读。
- 同一 SKU + 同一渠道的锁定时间段不允许重叠。
- 锁定期影响渠道需求计划格子是否可编辑。

## 3. 代码检索结论

### 3.1 本地持久化

现有持久化入口：

- `server/localDb.js`
- `server/local-data/db.json`

`localDb.js` 已提供：

- `readDb()`：读取缓存或 JSON 文件。
- `updateDb(updater)`：修改并写回 JSON。
- `nextId(rows)`：按数组生成自增 ID。
- `nowIso()`：生成当前 ISO 时间。
- `ensurePlatformStructures(db)`：补齐现有结构。

当前 `db.json` 已按区域组织：

- `system`：组织、用户、角色、权限。
- `master`：渠道、SKU、SPU、仓库、经销商、日历等主数据。
- `biz`：订单、库存、渠道经营等业务数据。
- `platform`：字典、导入导出、通知、流程等平台数据。

本功能适合落在 `biz` 下，避免污染主数据与平台配置。

### 3.2 渠道与 SKU 来源

可复用主数据：

- 二级渠道：`db.master.channel`，字段包括 `channel_code`、`channel_name`、`level`、`parent_code`、`status`。
- SKU：`db.master.sku`，字段包括 `sku_code`、`sku_name`、`category_code`、`lifecycle_status`、`status`。
- 三级品类：可通过 `db.master.category` 和 SKU 的 `category_code` 关联。

可复用现有渠道经营接口：

- `server/channelDealerOps.js` 已在 `/api/channel-dealer-ops/options` 中返回渠道、SKU、经销商、仓库等选项。
- 该模块已有 `ensureChannelDealerOpsStructures(db)` 和按业务数组补齐数据的模式，可作为新模块参考。

### 3.3 路由与权限

现有前端路由：

- `src/router/index.ts`
- 业务运营区已有 `/channel-dealer-ops`、`/intelligent`、`/inventory-ops` 等路由。

现有菜单：

- `src/data/webSidebarConfig.ts`
- “业务运营”分组中已有渠道与经销商经营中心。

现有后端权限拦截：

- `server/index.js` 的 `API_PERMISSION_RULES`
- 目前 `/api/channel-dealer-ops/*` 绑定到 `/channel-dealer-ops` 权限。

建议新增独立页面与权限：

- 页面路由：`/demand/channel-plan`
- API 前缀：`/api/demand/channel-plan`
- 权限路径：`/demand/channel-plan`
- 页面名称：`渠道需求计划`
- 权限编码：`biz:demand:channel-plan:view`

原因：该功能是独立计划业务，不只是渠道经营中心里的一个经营看板。

### 3.4 财年周与业务日历

现有页面与接口：

- `src/views/MdmCalendar.vue`
- `/api/master/calendar`
- `/api/master/calendar/month`

当前 `db.master.calendar` 样本为空，但字段设计中已出现 `fscl_year`、`fscl_week_range`。

本阶段建议：

- 版本主数据仍保存 `begin_week`、`end_week`、`plan_week` 字符串，例如 `2026W18`。
- 后端提供独立周工具函数，先按 ISO 周一至周日生成日期范围。
- 如果后续 `master.calendar` 有数据，再优先用日历表解析财年周。

### 3.5 Excel 能力

现有依赖：

- 前端和后端均已安装 `xlsx`。
- `server/index.js` 已有导出任务、SKU 导出、经销关系导出逻辑。
- `src/views/ExportTaskCenter.vue` 已有前端生成 Excel 下载方式。

建议渠道需求计划的 Excel 下载先直接在后端生成 workbook buffer 返回，或返回 JSON 让前端 `xlsx.writeFile` 生成。为了贴近 PRD 的接口语义，推荐后端返回 `xlsx` 文件流。

## 4. 本地 JSON 数据模型

本阶段在 `db.biz` 下新增以下数组。命名用业务语义，后续接库时可迁移到 PRD 的表。

```json
{
  "biz": {
    "channel_demand_plans": [],
    "channel_demand_plan_channels": [],
    "channel_demand_plan_skus": [],
    "channel_demand_plan_versions": [],
    "channel_demand_plan_channel_statuses": [],
    "channel_demand_plan_data": [],
    "product_lock_rules": [],
    "downstream_demand_plan_jobs": []
  }
}
```

### 4.1 `channel_demand_plans`

映射 PRD 的计划主表。

关键字段：

- `id`
- `plan_code`
- `plan_name`
- `plan_type`: `1=周度`，`2=月度`，本阶段主做周度。
- `create_type`: `1=手动`，`2=滚动自动`。
- `week_count`
- `channel_scope`: `1=全部渠道`，`2=指定渠道`。

- `created_by`、`created_time`、`updated_by`、`updated_time`

字段：

- `id`
- `plan_code`

指定 SKU 范围时使用。

字段：

- `plan_code`

- `sku_name`

### 4.4 `channel_demand_plan_versions`

映射 PRD 的版本表。

字段：

- `version_code`
- `version_label`
- `begin_week`
- `end_week`
- `week_count`
- `status`
- `last_version_code`

### 4.5 `channel_demand_plan_channel_statuses`

每个版本、每个渠道一条状态。

字段：

- `id`
- `plan_code`
- `version_code`
- `lv2_channel_code`

- `submit_status`: `0=未提交`，`1=已提交`

核心数据表，一行表示一个格子。

字段：

- `lv2_channel_code`
- `lv2_channel_name`

- `lv3_category_name`

- `plan_value`: `null` 或数字。
- `is_locked`
- `lock_rule_id`
- `lock_reason`
- `is_modified`

### 4.7 `product_lock_rules`

产品锁定期规则。

状态不直接手填，由当前日期和起止日期派生展示；落库字段用于快照与筛选。

### 5.4 计划接口

接口前缀使用 `/api/demand/channel-plan`。

计划列表：

- `GET /api/demand/channel-plan`
- 支持 `keyword`、`status`、`planType`、`createType`、分页。
- 返回计划及版本数量、最新版本、渠道数量、SKU 数量。

新增计划：

- `POST /api/demand/channel-plan`
- 校验计划名称、类型、创建方式、覆盖周数、渠道范围、SKU 范围。
- 生成 `plan_code`，格式建议 `CDP-YYYYMMDD-####`。
- 写入计划主数据、指定渠道范围、指定 SKU 范围。
- 不自动创建版本，前端可在创建后引导打开“新建首版”弹窗。

编辑计划：

- `PUT /api/demand/channel-plan/:planCode`
- 已确认计划不允许修改范围字段。
- 允许修改名称、备注、滚动配置等基础字段。

删除计划：

- `DELETE /api/demand/channel-plan/:planCode`
- 仅无版本或全草稿数据时允许逻辑删除。
- 本地 JSON 中建议设置 `status=-1`，不物理删除。

### 5.5 版本接口

版本列表：

- `GET /api/demand/channel-plan/:planCode/version`

新建版本：

- `POST /api/demand/channel-plan/:planCode/version`
- 参数：`version_code`、`version_label`、`begin_week`、`week_count`、`last_version_code?`。
- 生成周列表。
- 解析计划渠道范围和 SKU 范围。
- 初始化 `channel_statuses`。
- 初始化 `channel_demand_plan_data`。
- 如果有上一版本，重叠周继承 `plan_value`。
- 新增周的 `plan_value = null`。
- 创建时计算 `is_locked` 快照。

滚动生成版本：

- `POST /api/demand/channel-plan/:planCode/version/roll`
- 本阶段模拟 Quartz，不注册真实定时任务。
- 按上一版本 `begin_week + 1` 生成新版本。
- 自动生成版本号，建议 `VYYYYWww-RNN`。

版本详情：

- `GET /api/demand/channel-plan/version/:versionCode`
- 返回版本基础信息、周列表、渠道提交状态汇总、是否可编辑、是否可确认。

### 5.6 计划数据接口

查询数据：

- `GET /api/demand/channel-plan/version/:versionCode/data`
- 参数：`channelCode`、`keyword`、`categoryCode`、分页。
- 返回 SKU 行、周列、格子值、锁定信息、渠道提交状态。

保存数据：

- `PUT /api/demand/channel-plan/version/:versionCode/data`
- body 包含 `channelCode` 和批量格子：
  - `sku_code`
  - `plan_week`
  - `plan_value`
  - `force`
- 校验：
  - 版本已确认不可修改。
  - 渠道已提交不可修改。
  - 普通用户不可修改 `is_locked=true` 的格子。
  - 管理员可传 `force=true` 修改锁定格，并记录 `force_updated_by`、`force_updated_time`、`force_reason`。
  - `plan_value` 允许 `null` 和大于等于 0 的数字。

Excel 粘贴：

- 前端解析剪贴板为矩阵。
- 后端仍走批量保存接口，不单独设计导入接口。

### 5.7 渠道提交与撤回

提交渠道：

- `PUT /api/demand/channel-plan/version/:versionCode/submit`
- body：`channelCode`
- 如果存在空值，返回 `empty_count`，前端二次确认后带 `allowEmpty=true` 再提交。
- 提交后该渠道状态为已提交。
- 如果全部渠道已提交，版本状态自动变为待确认。

撤回渠道：

- `PUT /api/demand/channel-plan/version/:versionCode/withdraw`
- 仅管理员可操作。
- 版本状态为草稿或提交中时允许。
- 已确认不可撤回。
- 撤回后渠道恢复未提交，版本状态回退为提交中或草稿。

整体确认：

- `PUT /api/demand/channel-plan/version/:versionCode/confirm`
- 仅全部渠道提交后允许。
- 返回确认前汇总：
  - 渠道合计
  - SKU 合计
  - 空值数量
- 前端二次确认后带 `confirmed=true` 执行确认。
- 确认后版本状态为已确认，所有数据不可编辑。
- 写入 `downstream_demand_plan_jobs` 模拟 MQ 触发。

### 5.8 产品锁定期接口

列表：

- `GET /api/demand/channel-plan/product-lock-rules`
- 支持 SKU、渠道、状态、日期范围筛选。

新增：

- `POST /api/demand/channel-plan/product-lock-rules`
- 校验：
  - SKU 存在。
  - 渠道均为有效二级渠道。
  - 开始日期不可晚于结束日期。
  - 同 SKU + 同渠道时间段不可重叠。

编辑：

- `PUT /api/demand/channel-plan/product-lock-rules/:id`
- 仅未生效规则允许编辑。
- 生效中和已过期只读。

删除：

- `DELETE /api/demand/channel-plan/product-lock-rules/:id`
- 仅未生效规则允许删除。

刷新版本锁定快照：

- `POST /api/demand/channel-plan/version/:versionCode/rebuild-locks`
- 管理员操作。
- 用于锁定规则变更后主动刷新未确认版本的 `is_locked`。
- 已确认版本不刷新，保持确认时事实。

### 5.9 版本对比与导出

版本对比：

- `GET /api/demand/channel-plan/version/compare`
- 参数：`baseVersionCode`、`targetVersionCode`、`channelCode`。
- 返回并集周、并集 SKU，差异格标记 `changed=true`。
- 覆盖周期不同的周保留空值，用 `missing_side` 标明来源缺失。

Excel 下载：

- `GET /api/demand/channel-plan/version/:versionCode/export`
- 仅已确认版本开放。
- 输出字段：
  - 计划编号
  - 版本号
  - 二级渠道编码
  - 二级渠道名称
  - SKU 编码
  - SKU 名称
  - 财年周
  - 需求数量
- 空值按 PRD 输出为 `0`。
- 文件名：`渠道需求计划_{plan_code}_{version_code}_{yyyyMMdd}.xlsx`。

### 6.3 版本编辑表格

表格结构：

- 顶部渠道标签：二级渠道。
- 行：SKU。
- 列：版本周列表。
- 末列：合计。
- 单元格：`el-input-number` 或自定义紧凑输入。

交互规则：

- `plan_value=null` 显示为空，不显示 0。

- 管理员点击锁定格编辑时弹二次确认。
- 渠道已提交后整渠道只读。
- 版本已确认后全表只读。
- 支持从 Excel 复制粘贴到当前选中区域。

### 6.4 计划与版本流转

前端按钮规则：

- 提交中：未提交渠道可编辑，已提交渠道只读；管理员可撤回。
- 待确认：不可编辑数据，可整体确认。
- 已确认：只读，可下载 Excel、可参与对比。

状态展示：

- 使用 Element Plus `el-tag`。
- 状态映射：
  - 草稿：info
  - 提交中：warning
  - 待确认：primary
  - 已确认：success

## 7. 关键业务规则

### 7.1 周序列规则

输入 `begin_week = 2026W18`，`week_count = 8`。

输出：

- `2026W18`
- `2026W19`
- ...
- `2026W25`

跨年时自动进位。

本阶段日期范围：

- 先用 ISO 周规则计算 `week_start_date` 和 `week_end_date`。
- 如果以后 `db.master.calendar` 填充财年周映射，则优先用业务日历。

### 7.2 滚动继承规则

新版本周列表和上一版本周列表取交集。

- 交集周：按 `channel + sku + week` 继承上一版本 `plan_value`。
- 新增周：`plan_value = null`。
- 所有格子重新计算 `is_locked` 快照。
- 继承来的格子 `is_modified=false`，后续人工改动设为 `true`。

### 7.3 锁定快照规则

创建版本时根据 `product_lock_rules` 写入数据格：

- 如果 SKU 相同。
- 且渠道在锁定渠道列表中。
- 且计划周的日期范围与锁定期 `[start_date, end_date]` 有交集。
- 则 `is_locked=true`。

是否实时重算：

- 本阶段采用“创建版本时快照”。
- 规则变更后通过管理员手动刷新未确认版本快照。
- 已确认版本不刷新，避免影响历史事实。

### 7.4 权限规则

- 普通有权限用户：可维护未锁定、未提交、未确认的数据。
- 超级管理员：可强制修改锁定格，需二次确认并记录原因。
- 渠道撤回、刷新锁定快照由超级管理员操作。
- 已确认版本任何人不可修改。

### 7.5 下游触发规则

本阶段不接 MQ。

整体确认后：

- 写入 `db.biz.downstream_demand_plan_jobs`。
- 可选写入 `db.platform.notifications` 通知计划员。
- 后续分仓需求计划模块可读取该 job 或确认版本数据。

## 10. 风险与取舍

### 10.1 不接真实数据库的风险

本地 JSON 对并发写入不友好。当前项目本身也是本地持久化，本阶段可以接受，但批量保存时要减少写入次数，统一在一次 `updateDb` 中完成。

### 10.2 财年周口径风险

`master.calendar` 当前为空，因此财年周只能先按 ISO 周计算。若业务财年周和自然 ISO 周不一致，需要先补日历主数据或导入财年周映射。

### 10.3 Quartz 与 MQ 风险

当前技术栈没有 Quartz 和 MQ。规划中用：

- 手动滚动接口替代 Quartz。
- `downstream_demand_plan_jobs` 替代 MQ 消息。

后续接入真实任务调度和消息队列时，保留接口语义不变，只替换实现。

### 10.4 页面复杂度风险

渠道 × SKU × 8 周的矩阵容易变大。首版建议：

- SKU 分页。
- 只加载当前渠道。
- 批量保存当前页或变更格。
- 后续再做虚拟滚动。

### 10.5 权限风险

该功能涉及普通用户和管理员差异化编辑。必须同时修改：

- 前端路由 `permissionPath`
- 菜单配置
- 后端 `API_PERMISSION_RULES`
- 本地权限页与角色授权补齐

否则容易出现菜单可见但接口 403，或接口可用但菜单不可见。

## 11. 建议优先级

推荐先做阶段 1 + 阶段 2，形成可用闭环：

1. 计划管理。
2. 版本创建。
3. 表格填报。
4. 渠道提交。
5. 整体确认。

随后做产品锁定期，因为锁定期依赖版本数据格子存在。最后做滚动、对比、导出这些增强能力。
