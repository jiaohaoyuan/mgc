# 优化更改意见与阶段路线图

更新时间：2026-04-14  
适用范围：项目整体优化、排期评审、管理汇报、研发执行

## 1. 总体判断

当前项目已经具备：
- 前后端完整形态
- 平台基础能力
- 主数据管理能力
- 订单/库存/渠道/驾驶舱等业务页面基础
- 可运行、可联调、可继续扩展的代码底座

但如果目标是把它从“可运行原型”推进到“企业级可上线系统”，接下来优化重点不应再是单纯堆页面，而应转为三件事：

1. 统一事实源  
2. 补齐工程与安全底座  
3. 建立可交付、可运营、可治理的长期机制

## 2. 优化更改意见

### 2.1 数据层优化

#### 2.1.1 统一订单事实源

优化建议：
- 以 `biz.order_headers + biz.order_lines` 作为唯一订单主链
- `biz.orders` 仅保留短期兼容查询，不再新增写入
- 驾驶舱、渠道经营、库存联动统一切到新链

为什么：
- 现在同一业务对象存在双链路，最容易导致数据口径分裂

#### 2.1.2 统一库存事实层

优化建议：
- 以 `biz.inventory_ledger` 作为唯一库存事实层
- `biz.inventory_stock` 仅保留为展示汇总或缓存结果
- 锁定、释放、调拨、预警全部从 `ledger` 派生

为什么：
- 批次、效期、锁定、调拨这些核心动作都依赖台账级数据

#### 2.1.3 固化订单与库存桥接关系

优化建议：
- 将 `biz.inventory_locks` 定义为订单与库存唯一桥表
- 明确强约束：
  - `order_no -> order_headers`
  - `line_id -> order_lines`
  - `warehouse_code + sku_code + batch_no -> inventory_ledger`

为什么：
- 这是订单分配、锁定释放、库存核对的一致性中轴

#### 2.1.4 收敛经销授权主从关系

优化建议：
- 明确 `master.reseller_relation` 是授权主事实源
- `biz.channel_dealer_authorizations` 是经营镜像表
- 所有校验先走主数据，再同步经营域

为什么：
- 同一“授权”不能长期有两套真相

#### 2.1.5 收敛主数据与经营画像边界

优化建议：
- `master.reseller` 保留身份属性
- `biz.channel_dealer_profiles` 只保留经营扩展属性

建议边界：
- 主档：编码、名称、渠道归属、默认仓、基础合同属性
- 画像：负责人、信用等级、结算方式、经营标签

#### 2.1.6 建立全链路编码校验

优化建议：
- 对 `sku_code`、`warehouse_code`、`reseller_code`、`channel_code`、`org_code`、`product_code` 建立统一校验
- 导入、普通 CRUD、治理审批都前置校验

为什么：
- 这些编码就是系统真正的跨域主键

### 2.2 应用与架构层优化

#### 2.2.1 去掉前端 `mockData` 依赖

优化建议：
- 将部门、岗位、角色、用户、权限相关页面全部切到后端真实接口
- 前端保留类型定义，不再保留真实业务数据副本

为什么：
- 前端静态数据会让“页面状态”和“后端真实状态”分离

#### 2.2.2 去掉硬编码环境地址

优化建议：
- 所有前端接口地址统一走 `VITE_API_BASE`
- 文件上传地址也统一改为配置方式

为什么：
- 这是多环境部署和联调稳定性的最低要求

#### 2.2.3 降低技术栈不确定性

优化建议：
- 评估将 `vue: beta` 迁移到稳定版本
- 梳理依赖升级兼容性

为什么：
- 正式生产不宜长期依赖 Beta 版本

### 2.3 工程化优化

#### 2.3.1 建立最小自动化测试体系

优化建议：
- 前端：补关键页面/关键链路测试
- 后端：至少补核心服务层或 API 测试
- 将当前人工回归清单逐步转化为自动化用例

优先覆盖：
- 订单创建/提交/审核/分配
- 库存锁定/释放/调拨
- 主数据治理申请/审批/生效
- 导入任务生成与失败追溯

#### 2.3.2 建立 CI/CD 基线

优化建议：
- 至少补：
  - 安装依赖
  - 类型检查
  - 构建
  - 测试
  - 打包产物
- 后续再补自动部署与回滚

#### 2.3.3 建立多环境配置规范

优化建议：
- 增加 `.env.example`
- 区分本地、测试、预发、生产配置
- 敏感信息全部移出仓库

### 2.4 安全优化

#### 2.4.1 立即整改敏感配置

优化建议：
- 停止跟踪 `server/.env`
- 更换默认 JWT Secret
- 更换默认管理员口令
- 更换默认辅助动态码
- 清理仓库中的凭据文档或做脱敏处理

#### 2.4.2 建立配置与凭据管理规则

优化建议：
- 敏感配置只在部署环境注入
- 仓库中只保留模板和说明
- 增加变更记录与权限控制

### 2.5 运维与治理优化

#### 2.5.1 明确生产数据架构

优化建议：
- 明确 `local-data/db.json` 的角色
- 如果只用于演示，要明确标注
- 如果要走生产，应规划数据库、迁移、备份、恢复方案

#### 2.5.2 建立监控与告警基线

优化建议：
- 接口调用量
- 错误率
- 慢接口
- 导入导出任务失败率
- 库存预警数量
- 订单异常数量

#### 2.5.3 建立工作流统一对象映射

优化建议：
- 统一使用 `biz_type + biz_id`
- 通知、待办、审批、任务中心都按同一规则反查业务对象

## 3. 分阶段路线

## 阶段 0：止血与基线修复（1-2 周）

目标：
- 把当前最明显的安全、配置、环境问题先处理掉

本阶段必须完成：
- 清理仓库敏感配置
- 更换默认密钥/默认口令/默认辅助码
- 统一前端接口配置入口
- 明确本项目当前定位：演示版 / 试运行版 / 一期建设版
- 确认正式数据架构方向

阶段交付物：
- 安全整改记录
- 环境配置规范
- `.env.example`
- 项目定位说明

## 阶段 1：统一核心经营主链（2-4 周）

目标：
- 把订单、库存、桥接关系统一起来

本阶段重点：
- 统一订单主链到 `order_headers + order_lines`
- 统一库存事实层到 `inventory_ledger`
- 固化 `inventory_locks` 的桥接规则
- 调整驾驶舱、渠道经营、库存运营都基于统一口径

阶段交付物：
- 统一后的订单/库存数据链
- 页面联调通过记录
- 主链数据口径说明

## 阶段 2：统一主数据与经营镜像（3-5 周）

目标：
- 把主数据从“录入系统”变成“业务可靠底座”

本阶段重点：
- 明确 `reseller_relation` 与 `channel_dealer_authorizations` 主从关系
- 明确 `reseller` 与 `channel_dealer_profiles` 字段边界
- 建立主数据编码统一校验
- 关键主数据改为审批生效

阶段交付物：
- 主数据主从关系说明
- 编码校验规则
- 治理链与普通 CRUD 边界说明

## 阶段 3：补工程化与质量保障（3-4 周）

目标：
- 让后续开发不再高度依赖人工回归

本阶段重点：
- 建立最小自动化测试体系
- 补 CI/CD 基线
- 建立构建、测试、发布、回滚最小流程

阶段交付物：
- 自动化测试基线
- CI 流水线
- 发布与回滚说明

## 阶段 4：统一平台追踪域（2-4 周）

目标：
- 把通知、待办、审批、日志、任务都接到统一对象模型

本阶段重点：
- 统一 `biz_type + biz_id`
- 工作流去示例化、转真实业务驱动
- 导入导出任务、通知、日志、流程联动统一

阶段交付物：
- 平台对象映射规范
- 业务对象反查规则
- 统一待办/通知/审计链

## 阶段 5：补正式运维与交付治理（3-6 周）

目标：
- 让系统具备试点上线条件

本阶段重点：
- 部署架构
- 监控告警
- 备份恢复
- 灾备目标
- UAT 验收方案
- 培训方案
- 上线切换方案

阶段交付物：
- 运维手册
- 监控告警清单
- UAT 清单
- 上线切换方案

## 4. 推荐总顺序

如果按收益最大化排序，建议总顺序如下：

1. 阶段 0：先止血，解决安全与配置问题  
2. 阶段 1：统一订单和库存主链  
3. 阶段 2：统一主数据与经营镜像  
4. 阶段 3：补测试和发布底座  
5. 阶段 4：统一平台追踪域  
6. 阶段 5：补试点上线能力

## 5. 如果资源有限，最小可行路线

如果当前团队资源有限，建议最少先做以下三批：

### 第一批
- 安全整改
- 接口地址配置统一
- 项目定位明确

### 第二批
- 订单主链统一
- 库存事实层统一
- 订单-库存桥表统一

### 第三批
- 主数据授权主从统一
- 编码校验统一
- 最小自动化测试建立

## 6. 里程碑建议

| 里程碑 | 目标 |
| :--- | :--- |
| M1 | 安全、配置、环境基线完成 |
| M2 | 订单/库存主链口径统一 |
| M3 | 主数据与授权主从关系明确 |
| M4 | 自动化测试与 CI 基线可用 |
| M5 | 工作流/通知/日志统一映射 |
| M6 | 具备试点上线条件 |

## 7. 最终建议

如果要一句话概括后面的路线：

**先把系统从“能跑”改成“口径一致”，再把它从“口径一致”改成“可稳定交付”，最后再把它从“可交付”改成“可正式运营”。**
