# 接入 MySQL 与 Redis 前置检查清单

更新时间：2026-04-14  
适用范围：当前项目从本地 `JSON` 持久化逐步过渡到 `MySQL + Redis` 的准备阶段  
适用对象：项目负责人、后端开发、实施负责人、测试负责人、运维负责人

---

## 1. 文档目的

这份文档的目的，不是讨论“数据库好不好”，而是回答两个更实际的问题：

1. 现在这个项目，是否已经适合正式接入 `MySQL` 和 `Redis`。
2. 在正式切换前，还必须补齐哪些前置动作，避免后面出现数据错乱、链路断裂、上线失败、回滚困难的问题。

这份清单适合作为：

- 项目评审前置材料
- 技术方案评审检查表
- 接库启动会执行清单
- 上线前验收口径

---

## 2. 当前状态判断

结合当前项目现状，可以得出一个非常明确的结论：

**现在可以开始做“数据库接入准备”和“数据库连通验证”，但还不适合立即把主业务正式切换到 MySQL 与 Redis。**

原因很直接：

1. 项目已经存在 MySQL 和 Redis 的基础连接文件，但主业务读写仍大量依赖本地 `JSON`。
2. 订单、库存、授权、主数据治理这些核心链路还没有完成统一的数据访问层抽象。
3. 事务边界、迁移脚本、对账机制、回滚机制、自动化测试还没有补齐。
4. 当前环境配置和安全基线还不够稳，直接切真库会放大风险。

所以当前最合理的判断是：

- **可以做**：连通、建库、建表、验证、适配层准备、迁移脚本准备、低风险模块先试接。
- **不建议立即做**：把订单、库存、调拨、审批等主链路一次性改成只走 MySQL/Redis。

---

## 3. 当前项目现状摘要

### 3.1 已具备的基础

- 后端依赖中已经包含 `mysql2` 与 `redis`
- 已存在 `server/db/mysql.js`
- 已存在 `server/db/redis.js`
- 已存在部分初始化 SQL 文件
- 项目已经形成了比较完整的业务链和数据对象划分

### 3.2 当前仍然存在的关键问题

- 主服务真实业务仍以 `server/localDb.js` 为主要读写入口
- 大量接口仍直接 `readDb/updateDb`
- 数据访问逻辑没有统一抽象层
- 事务型业务动作尚未按照数据库事务口径梳理
- 当前测试能力不足，后端测试脚本仍是占位
- `.env` 已被项目跟踪，不适合直接进入正式数据库接入阶段
- 默认口令、默认辅助码、本地种子配置仍然存在

### 3.3 当前最主要的技术事实

简单说，当前系统是：

**业务模型和页面流程已经比较完整，但存储层仍偏本地文件模式，尚未完成数据库化改造的中间层准备。**

---

## 4. 总体判断口径

建议统一采用下面这句话作为项目口径：

**当前阶段适合启动数据库接入工程，不适合直接完成数据库切换。**

这个口径非常重要，因为它决定了推进方式。

正确推进方式应该是：

1. 先把接库前的准备动作补齐。
2. 再做 MySQL 和 Redis 的基础连通。
3. 再抽数据访问层。
4. 再做迁移和对账。
5. 最后按模块分批切换。

而不是：

1. 先把真库连上。
2. 再边改业务边补结构。

后者风险非常高。

---

## 5. 接入前总检查结论

建议先把接库前置条件分成 4 个等级：

### 5.1 A 类：必须完成，否则不要正式接库

- 事实源统一
- 表结构冻结
- 数据访问层抽象
- 事务边界定义
- 配置与密钥治理
- 迁移与回滚方案
- 核心链路回归清单

### 5.2 B 类：强烈建议完成，否则切库后维护成本高

- Redis 使用边界定义
- 关键索引设计
- 错误码与失败补偿机制
- 健康检查与监控
- 关键数据对账脚本

### 5.3 C 类：建议完成，可边切边补，但不要长期缺失

- 仓储层接口统一命名
- 环境模板和部署说明
- 慢查询与缓存策略
- 初始化数据装载机制

### 5.4 D 类：后续优化项

- 读写分离
- 分库分表
- 高级缓存穿透保护
- 复杂任务队列化

---

## 6. 正式接库前置检查清单

下面的清单建议项目逐项打钩，不满足就不要进入正式切库。

---

## 7. 第一类检查：业务口径与事实源是否统一

这一类是最重要的。如果业务口径不统一，数据库接得越早，历史包袱越大。

### 7.1 检查目标

确认每一类核心业务对象，只有一个主事实源。

### 7.2 必查对象

- 订单
- 订单明细
- 库存台账
- 库存锁定
- 库存流水
- 调拨单
- 经销授权
- 经销商经营档案
- 主数据变更申请
- 工作流待办

### 7.3 检查清单

- [ ] 订单主事实源是否明确为 `order_headers + order_lines`
- [ ] 旧订单链是否已明确为兼容链、分析链或废弃链
- [ ] 库存主事实源是否明确为 `inventory_ledger`
- [ ] 锁定记录是否明确为 `inventory_locks`
- [ ] 库存变更流水是否明确为 `inventory_transactions`
- [ ] 经销授权主事实源是否明确
- [ ] 主数据申请、版本、质检、冲突是否已明确主对象
- [ ] 各驾驶舱统计口径是否基于同一事实源

### 7.4 验收标准

- 每个核心对象都能说清楚“主表是谁、派生表是谁、兼容表是谁”
- 不存在两个页面分别维护两套真实业务数据
- 不存在“一个页面看 A 数据，另一个页面改 B 数据”的情况

### 7.5 未完成风险

- 接库后会出现同一业务对象两套数据并存
- 看板和明细口径不一致
- 后续迁移脚本无法确定真正迁哪一份数据

---

## 8. 第二类检查：数据库表结构是否冻结

接库前，不要求一次设计到完美，但必须达到“可以稳定建表”的程度。

### 8.1 检查目标

确认核心业务对象的字段、状态、主键、唯一键、关联键已经稳定。

### 8.2 必查内容

- 主键设计
- 唯一键设计
- 状态字段设计
- 时间字段设计
- 逻辑删除字段
- 审计字段
- 关联字段命名

### 8.3 检查清单

- [ ] 所有核心表都有明确主键
- [ ] 业务单号字段已明确唯一规则
- [ ] 状态字段已形成枚举清单
- [ ] 创建时间、更新时间字段统一
- [ ] 审核人、创建人、更新人字段统一口径
- [ ] 逻辑删除是否统一使用 `status` 或 `deleted_flag`
- [ ] 关联字段命名是否统一，比如 `order_no`、`sku_code`、`warehouse_code`
- [ ] 时间范围字段是否统一，如 `begin_date`、`end_date`

### 8.4 特别注意

以下对象不建议边接库边改字段：

- 订单头
- 订单行
- 库存台账
- 调拨单
- 经销授权
- 主数据变更申请

### 8.5 验收标准

- 数据库建表后，核心字段在一个版本内不再频繁改名
- 页面、接口、数据库字段之间能做到稳定映射

### 8.6 未完成风险

- 建表刚完成就反复改结构
- 迁移脚本频繁作废
- 前后端字段映射不断返工

---

## 9. 第三类检查：数据访问层是否抽象完成

这是切库前的关键工程动作。

### 9.1 检查目标

不能让业务代码继续大面积直接读本地 `JSON`，也不能让未来直接大面积写裸 SQL。

必须先抽一层统一的数据访问方式。

### 9.2 当前问题

当前大量业务代码直接依赖：

- `readDb()`
- `updateDb()`

这意味着未来如果直接换库，会变成全项目级别的硬切。

### 9.3 目标状态

建议至少形成下面这种分层：

1. 控制层：接收请求、返回结果
2. 业务层：执行业务规则
3. 仓储层：负责真实数据读写
4. 驱动层：本地 JSON / MySQL / Redis

### 9.4 检查清单

- [ ] 是否已定义统一的仓储层目录
- [ ] 订单模块是否已从 `readDb/updateDb` 抽离
- [ ] 库存模块是否已从 `readDb/updateDb` 抽离
- [ ] 主数据模块是否已从 `readDb/updateDb` 抽离
- [ ] 平台模块是否已具备最小统一读写接口
- [ ] 是否支持后续通过开关切换本地存储与 MySQL

### 9.5 验收标准

- 业务逻辑层不直接依赖底层存储实现
- 同一业务动作可以在不改业务规则的情况下替换存储介质

### 9.6 未完成风险

- 一旦切库，几十上百处代码同时改
- 容易出现局部漏改、混合读写、线上数据错乱

---

## 10. 第四类检查：事务边界是否定义完成

只要涉及订单、库存、调拨、审核，就一定有事务问题。

### 10.1 检查目标

明确哪些业务动作必须“要么一起成功，要么一起失败”。

### 10.2 必须梳理的动作

- 创建订单
- 提交审核
- 审核通过
- 审核驳回
- 自动分配
- 库存锁定
- 库存释放
- 调拨审核
- 调拨调出确认
- 调拨调入确认
- 主数据审批通过并生效

### 10.3 检查清单

- [ ] 创建订单时，订单头与订单行是否必须同事务提交
- [ ] 自动分配时，分配方案、库存锁定、异常生成是否必须同事务
- [ ] 驳回订单时，状态回退和锁释放是否必须同事务
- [ ] 调拨时，单据状态和库存变化是否已定义一致性规则
- [ ] 主数据审批通过时，版本记录和正式生效是否已定义事务边界
- [ ] 是否定义了失败补偿或重试逻辑

### 10.4 验收标准

- 每个关键业务动作都能说清楚事务边界
- 不存在“订单状态改了，但库存没改”这种半成功情况

### 10.5 未完成风险

- 真库上线后最常见的就是脏数据和半成功数据

---

## 11. 第五类检查：MySQL 使用边界是否明确

不要一接库就把所有东西都丢进去，先明确 MySQL 负责什么。

### 11.1 MySQL 建议承接的内容

- 主数据主表
- 订单主表与明细表
- 库存台账与流水
- 调拨单与轨迹
- 审批记录
- 经营分析需要沉淀的事实数据
- 平台配置、日志、任务主记录

### 11.2 MySQL 不建议一开始就做得很复杂的内容

- 复杂读写分离
- 分库分表
- 过早分域拆库
- 大规模历史归档自动化

### 11.3 检查清单

- [ ] 是否已完成建库方案
- [ ] 是否已完成命名规范
- [ ] 是否已完成字符集、时区、排序规则方案
- [ ] 是否已完成连接池参数方案
- [ ] 是否已完成索引初稿
- [ ] 是否已完成慢查询识别口径

### 11.4 验收标准

- 至少具备“能稳定建库建表、能稳定连通、能稳定执行业务 SQL”的能力

---

## 12. 第六类检查：Redis 使用边界是否明确

Redis 很容易被乱用，尤其是在项目早期。

### 12.1 正确的 Redis 角色

建议 Redis 先只做下面几类事情：

- 验证码
- 限流
- 登录会话或短期状态
- 看板缓存
- 未读数缓存
- 临时任务状态

### 12.2 不建议 Redis 一开始承接的内容

- 订单主事实源
- 库存主事实源
- 经销授权主事实源
- 主数据正式版本数据

### 12.3 检查清单

- [ ] 是否明确 Redis 仅用于缓存和短期状态
- [ ] 是否明确缓存失效策略
- [ ] 是否明确缓存键命名规范
- [ ] 是否明确缓存更新时机
- [ ] 是否明确缓存失败时的回退策略
- [ ] 是否明确不能把 Redis 当主库使用

### 12.4 验收标准

- 关闭 Redis 后，主业务仍然能跑，只是性能或体验下降

### 12.5 未完成风险

- 缓存和主数据口径不一致
- Redis 一失效，主业务直接不可用

---

## 13. 第七类检查：数据迁移与初始化方案是否具备

这是从本地 `JSON` 转到数据库最现实的问题。

### 13.1 检查目标

必须有一套明确的初始化迁移方案，而不是上线当天手工导一次。

### 13.2 必须准备的内容

- 初始化迁移脚本
- 增量回填策略
- 数据清洗规则
- 唯一键冲突处理规则
- 迁移日志
- 对账脚本

### 13.3 检查清单

- [ ] 是否已定义 `db.json -> MySQL` 的字段映射
- [ ] 是否已明确哪些数据需要迁移，哪些不迁移
- [ ] 是否已明确测试数据和真实数据隔离规则
- [ ] 是否已准备初始化导入脚本
- [ ] 是否已准备迁移后的核对脚本
- [ ] 是否已准备失败后的回滚方案

### 13.4 验收标准

- 同一份源数据，重复迁移结果一致
- 迁移前后主记录数量和关键口径可核对

### 13.5 未完成风险

- 迁移后丢单、丢库存、丢关系
- 迁移结果无法解释

---

## 14. 第八类检查：环境配置与密钥治理是否补齐

这一项必须在接真库前完成。

### 14.1 当前风险点

- `.env` 已纳入版本控制
- 存在默认数据库连接配置
- 存在默认账号口令
- 存在默认辅助动态码

### 14.2 检查清单

- [ ] `.env` 是否移出版本控制
- [ ] 是否提供 `.env.example`
- [ ] 是否区分本地、测试、预发、生产环境变量
- [ ] 是否更换默认数据库账号密码
- [ ] 是否更换默认 Redis 地址与认证方式
- [ ] 是否清理默认管理员口令
- [ ] 是否清理默认辅助动态码
- [ ] 是否清理真实第三方密钥明文

### 14.3 验收标准

- 仓库中不再出现真实密钥、真实口令、默认敏感值

### 14.4 未完成风险

- 真库接上以后，安全风险立即升级

---

## 15. 第九类检查：测试与回归能力是否具备

接库不是只是“连上就行”，而是要证明改完没把主链打断。

### 15.1 最低测试要求

至少要覆盖下面这些链路：

- 登录
- 用户/角色/权限
- 主数据增删改查
- 主数据治理申请与审批
- 订单创建
- 订单提交审核
- 订单审核通过/驳回
- 自动分配
- 库存锁定/释放
- 调拨流程
- 渠道授权与价格审批
- 驾驶舱核心统计读取

### 15.2 检查清单

- [ ] 是否已整理关键回归用例
- [ ] 是否已区分冒烟回归与全量回归
- [ ] 是否已准备接库后的联调脚本
- [ ] 是否已准备迁移后的数据核对清单
- [ ] 是否已准备至少一轮数据库切换演练

### 15.3 验收标准

- 接库前后，同一套关键业务操作得到的业务结果一致

### 15.4 未完成风险

- 切库成功了，但业务跑不起来

---

## 16. 第十类检查：上线、回滚、切换方式是否明确

不要等快上线了才讨论怎么切。

### 16.1 推荐方式

建议采用下面的切换策略：

1. 本地 JSON 保留一段时间作为兜底参考。
2. 先低风险模块切库。
3. 再主数据域切库。
4. 最后订单与库存主链切库。
5. 必要时短期双写或双读校验。

### 16.2 检查清单

- [ ] 是否明确模块切换顺序
- [ ] 是否明确是否需要双写
- [ ] 是否明确回滚到本地存储的条件
- [ ] 是否明确回滚操作步骤
- [ ] 是否明确切换窗口时间
- [ ] 是否明确切换后的观察期

### 16.3 验收标准

- 即使切换失败，也能在可控时间内退回旧方案

### 16.4 未完成风险

- 一旦切库失败，系统无路可退

---

不建议一刀切。建议按风险由低到高推进。

### 第一批：低风险平台模块

- 字典中心

### 第三批：主数据治理域

- 版本
- 质检

### 第四批：渠道经营域

- 经销商档案

## 18. 正式切库前的红线

下面这些情况，只要还有 1 条没处理，就不建议正式切主业务。

- 还说不清订单到底以哪张表为准
- 还说不清库存到底以哪张表为准
- 还没抽统一数据访问层
- 还没定义事务边界
- 还没准备迁移脚本
- 还没准备回滚方案
- 还没做关键链路回归
- 还保留默认密钥、默认口令、默认敏感配置

---

## 19. 接库前置验收口径

建议把下面这些作为“允许进入正式切库实施”的门槛：

- [ ] 主数据、订单、库存、渠道四大域事实源已明确
- [ ] 核心表结构已冻结
- [ ] 仓储层或数据访问层已完成第一版抽象
- [ ] MySQL 可稳定连通
- [ ] Redis 可稳定连通
- [ ] `.env` 与敏感配置治理完成
- [ ] 初始化迁移脚本可执行
- [ ] 对账脚本可执行
- [ ] 回滚方案可执行
- [ ] 核心业务回归通过

如果这 10 条没有全部满足，建议继续处于“接库准备阶段”，不要进入“主业务切换阶段”。

---

## 20. 推荐执行顺序

建议按下面顺序推进：

1. 先做配置与安全治理
2. 再做事实源统一与表结构冻结
3. 再抽数据访问层
4. 再补迁移脚本和对账脚本
5. 再接 MySQL 和 Redis 健康检查
6. 再切低风险模块
7. 最后切订单与库存主链

---

## 21. 一句话结论

**现在开始准备接入 MySQL 和 Redis 是合适的，但现在立即把主业务正式切到 MySQL 和 Redis 还偏早。**

当前最正确的动作，不是直接切库，而是先把：

**事实源、表结构、访问层、事务边界、迁移方案、回滚方案、测试回归、安全配置**

这 8 件事补齐。

只有这些前置条件完成后，数据库接入才会是一次受控升级，而不是一次高风险赌博。
