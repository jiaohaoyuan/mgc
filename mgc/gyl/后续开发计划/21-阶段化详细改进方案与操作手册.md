## 1. 全局执行原则

在所有阶段开始前，建议统一遵守以下原则：

### 1.1 先收敛事实源，再做新功能

不要一边保留旧链，一边继续往旧链上堆功能。  
后续所有新增逻辑，统一优先接入：
- 订单：`biz.order_headers + biz.order_lines`
- 库存：`biz.inventory_ledger`
- 授权：`master.reseller_relation`
- 平台追踪：`biz_type + biz_id`

### 1.2 一次只定义一个“唯一真相”

对任何核心对象，都要明确：
- 谁是主事实源
- 谁是派生表
- 谁只是兼容视图

否则页面再多，系统也会越来越乱。

### 1.3 每个阶段必须留下“治理资产”

不是只改代码，还要留：
- 数据口径说明
- 接口说明
- 联调清单
- 回归清单
- 验收清单

### 1.4 大文件先“止损”，再重构

当前后端入口 `server/index.js` 很大，后续不要一上来做大重构。  
建议策略：
- 第一阶段只做“收口”
- 第二、三阶段再逐步拆服务

## 2. 阶段 0：止血与基线修复

### 2.1 阶段目标

先解决当前最危险、最容易拖累后续所有开发的问题：
- 敏感信息风险
- 环境配置混乱
- 本地演示模式和正式运行模式混用
- 前端接口地址写死
- 项目定位不清

这一阶段不追求业务新功能，只做“止血”和“基线统一”。

### 2.2 当前问题

结合现有代码，阶段 0 需要处理的主要问题有：
- `server/.env` 已被 Git 跟踪
- 存在默认 JWT Secret 回退值
- 种子数据里存在默认管理员口令、默认辅助动态码
- 根目录存在凭据文档
- 前端多个页面写死 `http://localhost:3000/api`
- 当前后端实际运行依赖 `server/local-data/db.json`
- 项目尚未明确“演示模式”和“生产模式”边界

### 2.3 改进方案

#### 方案 A：做配置安全治理

核心目标：
- 敏感配置不再进入仓库
- 默认口令/默认密钥全部失效
- 本地环境和部署环境能独立配置

#### 方案 B：做环境统一治理

核心目标：
- 所有前端 API 地址统一从环境变量读取
- 所有后端关键运行参数统一从环境变量读取
- 明确当前默认运行模式是 `local-json demo mode` 还是 `production-like mode`

#### 方案 C：明确项目当前交付定位

核心目标：
- 写清楚这个项目当前到底是：
  - 演示版
  - 内部试运行版
  - 一期建设版

这是为了防止业务、研发、管理层对项目阶段理解不一致。

## 3. 阶段 1：统一核心经营主链

### 3.1 阶段目标

把系统最关键的三条经营链打通：
- 订单主链
- 库存主链
- 订单与库存桥接链

如果这一步不做，后面的渠道经营、驾驶舱、流程、报表都会继续建立在分裂口径上。

### 3.2 当前问题

当前核心问题有三类：

#### 问题 1：订单双事实源
- 旧链：`biz.orders`
- 新链：`biz.order_headers + biz.order_lines`

#### 问题 2：库存双口径
- 汇总口径：`inventory_stock`
- 事实口径：`inventory_ledger`

#### 问题 3：订单和库存的连接关系尚未被制度化
- 目前通过 `inventory_locks` 连接
- 但还需要明确它是唯一桥表

### 3.3 改进方案

#### 方案 A：统一订单事实源到新链

目标：
- 所有新增订单、审核、分配、履约都基于：
  - `order_headers`
  - `order_lines`

#### 方案 B：统一库存事实层到 `inventory_ledger`

目标：
- 所有精确库存业务动作统一基于台账
- `inventory_stock` 只做展示汇总

#### 方案 C：固化 `inventory_locks` 为桥表

目标：
- 订单和库存之间只认一条桥：
  - `order_no`
  - `line_id`
  - `warehouse_code + sku_code + batch_no`

### 3.4 如何操作

#### 步骤 1-1：冻结旧订单链新增写入

操作内容：
- 任何新增订单相关逻辑，不再写 `biz.orders`
- 旧接口如果还保留，只做兼容读取或兼容转换

建议操作顺序：
1. 检查所有订单写入入口
2. 确认只有 `orderPhase2.js` 继续承担正式写入
3. 标记旧链 `biz.orders` 为兼容数据源
4. 在旧页面中增加说明：该页面为兼容分析视图，不再承担真实订单创建

## 5. 阶段 3：补工程化与质量保障

### 5.1 阶段目标

把项目从“靠人工联调保障质量”推进到“有最小自动化保障”的状态。

### 5.2 当前问题

- 后端 `test` 仍是占位脚本
- 前端没有正式自动化测试体系
- 没有 CI/CD 流程
- 构建通过不代表业务回归通过

### 5.3 改进方案

#### 方案 A：补最小测试金三角

建议三类测试同时建立：
- 核心后端 API 测试
- 核心前端页面联动测试
- 关键业务主链回归测试

#### 方案 B：补 CI 基线

至少保证每次提交都能自动执行：
- 依赖安装
- 类型检查
- 构建
- 测试

#### 方案 C：补代码组织治理

建议逐步把大文件拆分，不要继续把新逻辑塞进 `server/index.js`

### 5.4 如何操作

#### 步骤 3-1：先定义测试优先级

优先测试链路：
1. 订单创建 -> 提交 -> 审核 -> 分配 -> 履约
2. 库存锁定 -> 释放 -> 调拨
3. 主数据申请 -> 审批 -> 生效
4. 导入任务 -> 错误明细 -> 任务中心

先从这几条主链做，不要一开始追求全覆盖。

#### 步骤 3-2：后端补 API 测试

建议先做：
- `orderPhase2.js`
- `inventoryOps.js`
- `mdmGovernance.js`
- `workflowCenter.js`

操作方式：
1. 建立测试目录
2. 准备演示数据/测试种子数据
3. 先覆盖最关键成功路径和失败路径

建议覆盖点：
- 创建订单成功
- 创建订单因授权失败
- 审核后自动分配
- 锁定库存后驳回释放
- MDM 停用前引用检查
- 工作流对象跳转路径生成

#### 步骤 3-3：前端补关键页面测试

建议先覆盖页面：
- `OrderClosedLoopCenter.vue`
- `InventoryOpsCenter.vue`
- `MdmGovernanceCenter.vue`
- `ImportTaskCenter.vue`
- `WorkflowCenter.vue`

建议验证内容：
- 页面加载
- 查询参数联动
- 列表与详情跳转
- 操作按钮权限
- 接口异常提示

#### 步骤 3-4：建立 CI 流程

至少补以下节点：
1. `npm install`
2. `npm run build`
3. 后端测试
4. 前端测试

如果资源有限，先实现“合并前必须过构建 + 核心测试”。

#### 步骤 3-5：逐步拆后端大文件

建议拆分方向：
- `server/index.js` 只保留：
  - 应用初始化
  - 中间件挂载
  - 路由注册
- 业务逻辑拆到：
  - `services`
  - `domain`
  - `repositories`

注意：
- 不要本阶段一次性重构太多
- 只拆新增和高频修改部分

### 5.5 风险与注意事项

- 不要一开始追求全量自动化
- 不要把测试建立成“只测最简单 happy path”
- 不要在未统一主链前做大量陈旧链测试

### 5.6 阶段交付物

- 核心 API 测试
- 关键页面测试
- CI 流程
- 最小发布门禁

### 5.7 验收标准

- 核心主链至少有自动化回归
- CI 能在提交时执行构建和测试
- 新改动不再只靠人工盲测

## 6. 阶段 4：统一平台追踪域

### 6.1 阶段目标

把通知、待办、审批、日志、导入导出任务接到同一套对象映射模型，让平台层真正成为“业务追踪层”。

### 6.2 当前问题

- `workflow_*` 既有真实业务对象，也有示例数据
- `notifications`、`operation_logs`、`import_tasks`、`export_tasks` 的对象关联口径还不完全统一
- 有的平台记录能反查业务对象，有的不能

### 6.3 改进方案

#### 方案 A：统一软链接规则

统一规则：
- 用 `biz_type + biz_id` 表示业务对象

#### 方案 B：建立统一平台事件出口

任何关键业务动作，都统一经过一个平台事件封装层，负责：
- 写通知
- 写待办
- 写审批
- 写日志
- 写任务

#### 方案 C：工作流去示例化

把工作流中心从“演示待办中心”逐步变成“真实业务待办中心”。

### 6.4 如何操作

#### 步骤 4-1：定义统一业务对象字典

建议先明确以下对象类型：
- `ORDER`
- `TRANSFER`
- `MDM`
- `TASK`
- `IMPORT_TASK`
- `EXPORT_TASK`
- `EXCEPTION`

每个对象都明确：
- `biz_type`
- `biz_id`
- 默认跳转路径
- 默认标题规则

#### 步骤 4-2：统一生成链接规则

当前 `workflowCenter.js` 已有 `buildBizLinkPath` 雏形，建议扩展为平台统一工具。

操作方式：
1. 把链接生成规则抽离
2. 通知、待办、审批、消息全部调用同一个方法
3. 前端跳转统一按同一规则处理

建议涉及文件：
- [`server/workflowCenter.js`](C:/Users/Administrator/Desktop/mgc/mgc/gyl/server/workflowCenter.js)
- [`server/index.js`](C:/Users/Administrator/Desktop/mgc/mgc/gyl/server/index.js)

#### 步骤 4-3：建立统一平台事件封装

建议把这些动作统一封装：
- `appendOperationLog`
- `appendSecurityLog`
- 通知创建
- 审批待办创建
- 任务记录创建

操作方式：
1. 新建平台事件工具模块
2. 各业务模块不要再各自拼日志/通知结构
3. 统一字段标准

建议优先接入模块：
- `orderPhase2.js`
- `inventoryOps.js`
- `mdmGovernance.js`
- 导入导出逻辑

#### 步骤 4-4：把真实业务事件接入工作流

优先接入事件：
- 订单提交审核
- 主数据申请提交
- 调拨单提交审批
- 导入失败
- 异常待处理

操作方式：
1. 明确每个事件对应的待办类型
2. 明确处理人来源
3. 明确跳转路径
4. 明确完成动作后如何关闭待办

#### 步骤 4-5：去掉不必要的演示性待办注入

`seedWorkflowDataForUser` 可保留在演示模式，但正式模式应关闭。

建议处理方式：
- 增加运行模式开关
- 演示环境允许注入
- 真实环境只显示真实业务生成的待办

### 6.5 风险与注意事项

- 不要让每个模块继续各自维护一套通知结构
- 不要一边用真实业务，一边还混大量示例待办
- 不要让同一业务对象在不同平台表里用不同 `biz_type`

### 6.6 阶段交付物

- 平台对象映射字典
- 统一链接生成规则
- 平台事件封装层
- 工作流真实接入清单

### 6.7 验收标准

- 通知、待办、审批、消息能统一反查业务对象
- 关键业务动作都会产生对应平台记录
- 工作流中心不再主要依赖示例数据

## 7. 阶段 5：补正式运维与交付治理

### 7.1 阶段目标

让系统从“研发视角可运行”变成“企业视角可试点上线”。

### 7.2 当前问题

- 缺部署方案
- 缺监控与告警
- 缺备份恢复方案
- 缺试点上线流程
- 缺 UAT、培训、推广、切换方案

### 7.3 改进方案

#### 方案 A：明确正式运行架构

要明确：
- 是否继续本地 JSON
- 是否切回正式数据库
- 是否使用 Redis
- 是否需要独立前后端部署

#### 方案 B：建立运维监控体系

至少补：
- 应用监控
- 接口监控
- 任务监控
- 数据风险监控

#### 方案 C：建立试点上线治理包

至少补：
- UAT
- 培训
- 切换
- 回滚
- 试点组织范围

### 7.4 如何操作

#### 步骤 5-1：明确正式部署方式

建议先做一个选择决策：

方案 1：短期试点部署
- 允许简化部署
- 但必须有备份与恢复

方案 2：正式长期部署
- 必须恢复数据库架构
- 必须补迁移、监控、告警、部署规范

不建议长期使用“本地 JSON + 直接运行”的方式作为正式生产模式。

#### 步骤 5-2：建立部署与启动文档

文档至少要写清楚：
- 前端怎么部署
- 后端怎么部署
- 环境变量有哪些
- 启动顺序是什么
- 健康检查怎么看
- 出错怎么回滚

#### 步骤 5-3：建立监控指标基线

至少包含以下指标：
- 接口调用量
- 错误率
- 慢接口
- 导入导出失败率
- 工作流任务失败率
- 库存预警数量
- 订单异常数量
- 登录失败次数

#### 步骤 5-4：建立备份与恢复流程

如果继续使用文件模式：
- 备份 `db.json`
- 定期快照
- 恢复演练

如果切数据库模式：
- 全量备份
- 增量备份
- 恢复脚本
- 结构迁移脚本

#### 步骤 5-5：建立 UAT 验收方案

建议按角色验收：
- 管理员
- 业务下单人员
- 审批人员
- 仓配人员
- MDM 管理员
- 管理层查看人员

建议按主链验收：
1. 订单主链
2. 库存主链
3. 主数据治理链
4. 平台追踪链

#### 步骤 5-6：建立培训与试点上线方案

建议内容：
- 试点部门
- 试点角色
- 上线时间
- 培训计划
- 现场支持人
- 故障升级路径

#### 步骤 5-7：建立回滚方案

必须明确：
- 什么情况下回滚
- 回滚到哪个版本
- 数据如何回滚
- 通知谁
- 谁批准回滚

### 7.5 风险与注意事项

- 不要把“会部署”当成“能上线”
- 不要没有 UAT 就直接试点
- 不要没有回滚预案就推动真实业务切换

### 7.6 阶段交付物

- 部署文档
- 监控告警清单
- 备份恢复方案
- UAT 验收清单
- 培训方案
- 试点上线方案
- 回滚方案

### 7.7 验收标准

- 新环境可按文档独立部署
- 关键指标有监控
- 有备份恢复能力
- 有 UAT 记录
- 有培训和试点上线方案
- 出问题时有明确回滚路径

## 8. 推荐整体推进顺序

### 8.1 标准顺序

1. 阶段 0：先止血  
2. 阶段 1：统一经营主链  
3. 阶段 2：统一主数据与镜像  
4. 阶段 3：补工程化  
5. 阶段 4：统一平台追踪域  
6. 阶段 5：补正式运维与交付治理

### 8.2 为什么这个顺序最合适

- 阶段 0 不做，后面所有阶段都带着安全和环境隐患推进
- 阶段 1 不做，后面所有分析和流程都会建立在错口径上
- 阶段 2 不做，后面渠道经营和治理体系会持续双轨
- 阶段 3 不做，后面每次改动都只能靠人工回归
- 阶段 4 不做，平台层会一直只是展示层，不是业务追踪层
- 阶段 5 不做，系统仍不具备真正试点上线条件
