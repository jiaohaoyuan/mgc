# 数据关系收敛优先级清单

更新时间：2026-04-14  
目标：明确“先收敛哪几条数据关系，才能最快提升系统一致性、可维护性和可上线性”

## 1. 排序原则

本清单按以下四个维度综合排序：
- 对经营口径影响是否大
- 是否会导致多页面数据不一致
- 是否会影响后续新功能接入
- 改造收益是否高于改造成本

优先级定义：
- `P0`：必须先收敛，否则系统核心口径长期分裂
- `P1`：一期必须收敛，否则新功能持续建立在不稳定底座上
- `P2`：二期前应收敛，主要影响治理与运维效率

## 2. P0：必须先收敛

### P0-1 订单事实源收敛

现状：
- 旧链：`biz.orders`
- 新链：`biz.order_headers + biz.order_lines`

问题：
- 同样是“订单”，系统内存在两套数据载体
- 智能订购旧页和订单闭环主链不在同一事实源上

建议目标：
- 明确 `order_headers + order_lines` 为唯一订单主链
- `biz.orders` 只允许短期兼容读取，不再新增写入
- 所有新接口、新页面、新分析统一改读新链

优先级原因：
- 这是全系统最大的口径分裂点
- 不先收敛，后面的库存、渠道、驾驶舱都无法真正统一

交付标准：
- 新订单只写新链
- 旧页兼容层可保留，但必须标记为兼容视图
- 驾驶舱、渠道经营、库存联动全部统一到新链

### P0-2 库存事实层收敛

现状：
- 汇总层：`inventory_stock`
- 事实层：`inventory_ledger`

问题：
- 订单分配、批次、锁定、临期判断应基于台账
- 汇总库存如果继续被业务判断直接使用，容易造成逻辑偏差

建议目标：
- 明确 `inventory_ledger` 为唯一库存事实层
- `inventory_stock` 仅保留为聚合视图/缓存结果
- 所有精确业务动作统一从 `ledger` 推导

优先级原因：
- 这是订单闭环和库存运营能否一致的关键
- 也是调拨、锁定、预警可信度的基础

交付标准：
- 锁定、释放、调拨、预警全部从 `ledger` 派生
- `inventory_stock` 只能用于展示，不参与关键决策

### P0-3 订单与库存桥接关系收敛

现状：
- 订单与库存依赖 `inventory_locks` 连接

问题：
- 如果 `order_no / line_id / warehouse_code / sku_code / batch_no` 这组关系不稳定，订单分配和库存台账就无法对账

建议目标：
- 把 `inventory_locks` 定义为订单-库存唯一桥表
- 明确字段级约束：
  - `order_no -> order_headers`
  - `line_id -> order_lines`
  - `warehouse_code + sku_code + batch_no -> inventory_ledger`

优先级原因：
- 这是订单和库存的一致性中轴

交付标准：
- 任一锁记录都能反查到订单行和库存批次
- 订单关闭/驳回/履约时锁定释放规则统一

## 3. P1：一期必须收敛

### P1-1 经销授权主从关系收敛

现状：
- 主数据授权：`master.reseller_relation`
- 经营域授权：`biz.channel_dealer_authorizations`

问题：
- 现在两边都有“授权”含义，容易长期双写、双判定

建议目标：
- 明确：
  - `reseller_relation` 是主事实源
  - `channel_dealer_authorizations` 是经营镜像/缓存/增强视图
- 所有授权校验先落主数据，再同步经营域

优先级原因：
- 这条链同时影响订单校验、渠道经营、风险扫描、驾驶舱

交付标准：
- 只允许一处主写入
- 另一处通过同步或派生生成
- 风险扫描与业务校验口径一致

### P1-2 经销商主档与经营画像收敛

现状：
- 主档：`master.reseller`
- 经营画像：`biz.channel_dealer_profiles`

问题：
- 经营画像中有不少字段来自主档延展，如果同步机制不明确，画像很容易过期

建议目标：
- 明确主档字段和经营扩展字段边界
- 主档负责身份属性，画像负责经营属性

字段建议：
- 主档负责：编码、名称、渠道归属、默认仓、合同基础属性
- 画像负责：负责人、信用等级、结算方式、经营标签

优先级原因：
- 这能减少“同一个经销商两套信息”的维护成本

### P1-3 主数据编码统一校验

对象：
- `sku_code`
- `warehouse_code`
- `reseller_code`
- `channel_code`
- `org_code`
- `product_code`

问题：
- 这些编码是全系统真正的跨域主键，一旦不一致，数据链全部断裂

建议目标：
- 建立统一编码校验规则
- 所有 CRUD、导入、治理审批都前置校验编码存在性和唯一性

优先级原因：
- 成本低、收益极高
- 是后续所有治理工作的底层前提

### P1-4 MDM治理链与普通 CRUD 收敛

现状：
- 一边有 `mdm_change_requests / versions / issues / conflicts`
- 一边普通 `/api/master/*` 还可以直接改主表

问题：
- 如果高风险主数据既能走治理，也能绕过治理，就会出现“制度在，执行不在”

建议目标：
- 关键主数据改造为：
  - 新增/修改/停用先走治理申请
  - 普通 CRUD 仅保留低风险字段
  - 高风险动作必须审批生效

优先级原因：
- 不收敛这一条，主数据质量很难稳定

## 4. P2：二期前应收敛

### P2-1 工作流业务对象映射收敛

现状：
- `workflow_todos / approvals / messages / tasks` 已支持 `biz_type + biz_id`
- 但仍有示例/种子逻辑

建议目标：
- 把 `biz_type + biz_id` 定义为平台域统一软链接
- 所有真实业务待办都统一映射到这一套机制

优先级原因：
- 影响流程协同体验，但不先于订单/库存事实源

### P2-2 通知、日志、任务中心对象映射收敛

对象：
- `notifications`
- `operation_logs`
- `import_tasks`
- `export_tasks`

建议目标：
- 对每一类平台记录统一定义业务对象映射规则
- 保证能通过 `biz_type / biz_id / biz_object_type / biz_object_id` 稳定反查业务单据

### P2-3 配置版本与运行配置收敛

对象：
- `system_configs`
- `system_config_versions`

建议目标：
- 所有关键算法权重、阈值、流程参数统一走版本化
- 避免页面当前值和实际生效值脱节

### P2-4 管理报表与实时分析口径收敛

对象：
- `management_reports`
- 驾驶舱实时聚合接口

建议目标：
- 明确“报表快照”和“实时指标”不是一回事
- 对外汇报必须注明口径和时间点

## 5. 推荐实施顺序

### 第一批：先打通经营主链

1. `order_headers/order_lines` 统一订单主链  
2. `inventory_ledger` 统一库存事实层  
3. `inventory_locks` 固化订单-库存桥关系

### 第二批：再统一主数据与经营镜像

1. `reseller_relation` 与 `channel_dealer_authorizations` 主从化  
2. `reseller` 与 `channel_dealer_profiles` 字段边界清晰化  
3. 主数据编码全链校验化

### 第三批：最后统一平台追踪域

1. `biz_type + biz_id` 成为工作流/通知统一软链接  
2. 任务、日志、通知稳定反查业务对象  
3. 报表快照和实时指标明确区分

## 6. 建议的责任划分

| 收敛主题 | 推荐牵头角色 |
| :--- | :--- |
| 订单事实源 | 业务后端负责人 |
| 库存事实层 | 仓配/库存模块负责人 |
| 主数据授权主从 | MDM负责人 |
| 编码统一校验 | 平台基础负责人 |
| 工作流对象映射 | 平台流程负责人 |
| 报表口径治理 | 数据产品/驾驶舱负责人 |

## 7. 最终建议

如果只能先做三件事，建议按这个顺序推进：

1. 先统一 `order_headers/order_lines`  
2. 再统一 `inventory_ledger`  
3. 然后明确 `reseller_relation` 是授权主事实源

原因很简单：
- 这三步一旦完成，系统的订单、库存、渠道三条主链就会第一次真正对齐
- 后面的工作流、通知、驾驶舱、治理、报表，都会在一个更稳定的底座上进行
